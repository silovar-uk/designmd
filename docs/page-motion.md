# ページモーション設計

> Status: beta  
> Scope: HTML版、長い読み物、教材、説明ページ

ページモーションは、要素を派手に動かすための装飾ではない。

> 読み手が、いつ気づき、どこへ視線を移し、いつ止まり、いつ笑い、どう内容へ戻るかを、時間と空間として編集する。

スマートフォンでは、登場順、間、スクロールによる回収が中心になる。PCでは、それに加えて、左右の距離、同時表示、余白、ポインター操作を使える。

端末が変わっても同じアニメーションを拡大・縮小して使わない。使える空間と入力方法が変われば、動きの役割も変える。

## 原則

1. 動きには編集上の役割を持たせる
2. 主要情報を動きだけに依存させない
3. 一つの大きな領域に、強い動きは1〜2回まで
4. 同じフェードを全セクションへ配らない
5. 動いた後は止める
6. オチの後は静かな状態へ戻る
7. `prefers-reduced-motion`を尊重する
8. 初期表示では完成前のDOMや旧スタイルを見せない
9. 端末ごとに、時間・空間・入力方法を設計し直す
10. 小さな操作反応と、強い編集演出を同じ速度で扱わない
11. 位置を動かす場合は、移動前後の関係を説明できること
12. 読み手がアニメーションの終了を待たないと操作できない状態をつくらない

全部を動かすと、全部が背景になる。かなり元気な背景である。

## 参照元の選び方

一つのサイトを完成見本にしない。参照元を三層へ分ける。

### 1. モーションの文法

モーションの目的、空間関係、速度、連続性を決めるための参照元。

#### Apple Human Interface Guidelines / Motion

https://developer.apple.com/jp/design/human-interface-guidelines/motion

借りる考え方：

- モーション自体を目的にしない
- フィードバックは短く正確にする
- 頻繁な操作へ余計な待ち時間を加えない
- 重要情報をモーションだけで伝えない
- 可能な限り、動作を中断・回避できるようにする

Appleの外見を借りるのではなく、「何のために動かすか」「利用者を待たせていないか」という判断基準を借りる。

#### Material Design / Material motion

https://m1.material.io/motion/material-motion.html

借りる考え方：

- モーションは空間関係、機能、意図を示す
- 画面間の連続性を保つ
- 読み手の焦点を次の対象へ導く
- 位置変化によって、情報の階層や所属を説明する

紙や物体の見た目を再現するのではなく、「どこから来て、どこへ移り、何と関係するか」を説明する文法として使う。

#### Carbon Design System / Motion

https://carbondesignsystem.com/elements/motion/overview/  
https://carbondesignsystem.com/elements/motion/choreography/

借りる考え方：

- productive motionとexpressive motionを分ける
- 日常的な操作は速く効率的にする
- 重要な転換や状態変化だけ、少し強く見せる
- 複数要素はばらばらに動かさず、構成として振り付ける
- 不自然なバウンド、伸縮、急停止を避ける

designmdでは、次のように読み替える。

- productive：ホバー、選択、開閉、進捗、コピー完了
- expressive：章の転換、比較の成立、反論、オチ、本文への帰還

### 2. 読み物としての編集

スクロール、記事固有の仕掛け、読み進める動機を考えるための参照元。

#### The Pudding / Storytelling

https://pudding.cool/process/how-to-make-dope-shit-part-3/  
https://pudding.cool/process/how-to-implement-scrollytelling/demo/rollyourown/

借りる考え方：

- スクロールを、単なる表示条件ではなく物語の拍として使う
- 一つの図や対象を保ちながら、状態だけを段階的に変える
- 各ステップの状態を明示し、高速スクロールでも破綻させない
- PCとスマートフォンの画面高を別々に確認する
- 実装コストと読解効果が釣り合う場所だけで使う

スクロール演出を使う条件：

1. 同じ対象を見続ける意味がある
2. 状態変化の順番が理解に必要
3. 静止した縦並びより関係を理解しやすい
4. 途中の状態を飛ばしても最終情報へ到達できる
5. モバイルとPCの双方で高さを検証できる

条件を満たさない場合、通常の縦スクロールを選ぶ。

#### Codrops / Editorial rhythm

https://tympanus.net/codrops/2026/03/27/pell-mell-crafting-a-visual-exploration-platform-with-editorial-rhythm/  
https://tympanus.net/codrops/2023/01/18/on-scroll-typography-animations/

借りる考え方：

- 画面全体を動かすより、小さな反応の集合でリズムをつくる
- 強いタイポグラフィと余白を、装飾ではなく構造として使う
- インターフェイスを前へ出しすぎず、内容を主役にする
- `IntersectionObserver`など軽い仕組みから始める
- 文字の動きは意味のある箇所へ限定する

デモの外見やライブラリ構成をそのまま移植しない。実験的な動きを、制作物の標準へ昇格させない。

#### オモコロ

- 景色のカラーコード  
  https://omocoro.jp/kiji/557009/
- ダブル  
  https://omocoro.jp/kiji/557544/
- オモコロ特集  
  https://omocoro.jp/kiji/

借りる考え方：

- 記事・題材固有の仕掛けを許す
- 読者の予想を一度だけ外す
- 本文、画像、操作を一つの企画として扱う
- 仕掛けが動かなくても本文は読める
- 笑いを単独で置かず、その直前の内容を受けて成立させる

借りないもの：

- 固有のロゴ、書体、キャラクター、配色
- タイトルやギャグの調子
- 題材と無関係な奇抜さ
- すべてのページを特殊サイトにする発想

### 3. 技術とアクセシビリティ

動きの停止、処理負荷、実装方法を決めるための参照元。

#### W3C / WCAG

https://www.w3.org/WAI/WCAG22/Understanding/animation-from-interactions  
https://www.w3.org/WAI/WCAG22/Techniques/css/C39  
https://www.w3.org/WAI/WCAG22/Techniques/client-side-script/SCR40

最低条件：

- `prefers-reduced-motion`をCSSとJavaScriptの双方で確認する
- 非本質的な移動は停止できる
- 動きを止めても情報と操作を失わない
- スクロールに連動する追加移動を必要以上に使わない
- ページ独自の「動きを減らす」操作を検討する

#### web.dev / Animation performance

https://web.dev/articles/animations-guide  
https://web.dev/articles/animations-overview

最低条件：

- `transform`と`opacity`を中心にする
- `width`、`height`、`top`、`left`の連続アニメーションを避ける
- レイアウトやペイントを毎フレーム発生させない
- `will-change`を常用しない
- DevToolsで実際の処理を確認する

## 五つの動き

### 1. 登場

新しい章、対象、視点が現れたことを知らせる。見出しと本文をわずかにずらし、初回だけ実行する。

PCでは、横から入れる前に、その横方向が情報の出所や次の位置と対応しているか確認する。

### 2. 引き継ぎ

前の要素から次の要素へ、関係や視点が移ったことを示す。線やラベルを動かす場合も、何を渡しているかを言葉で説明できることを条件にする。

PCでは左右の距離を使えるが、斜め方向へ無秩序に飛ばさない。同じグリッドや軸の上で受け渡す。

### 3. ため

直後の発見、反論、オチの前に、一呼吸ぶんの時間差をつくる。重要条件や安全情報を演出のために遅らせない。

PCでは空白や静止も「ため」として使う。動きを追加しなくても成立する。

### 4. オチ

予想を一度だけ外し、理解や記憶を強める。読者を笑いの対象にせず、すべての見出しをオチにしない。

PCでは大きな面積が使えるため、オチも大きくしがちになる。画面全体を揺らす前に、対象一つの変化で足りないか確認する。

### 5. 帰還

動きやユーモアの後で、本文へ戻ったことを知らせる。笑いが終わったのに、ページだけが踊り続けない。

PCでは、動いた要素を元のグリッド、文字サイズ、コントラストへ戻す。

## PCでの使い方

PC版を、スマートフォン版の横幅を広げたものとして作らない。

### 空間を文法として使う

- 左右の位置に、出所、順序、比較、因果の意味を持たせる
- 一つの対象が移動した場合、移動前後の連続性を残す
- 画面外から入れる場合、どこから来たかに意味を持たせる
- 周辺視野で複数要素を同時に動かさない
- 斜め方向へ交差する移動を避ける

### 同時表示を使う

PCでは、Before / After、根拠 / 判断、原材料 / 完成形を同時に置ける。

すべてを時間差で順番に見せず、同時に見せた方が比較しやすい場合は静止させる。動きは「比較が成立した瞬間」や「焦点を片方へ移す瞬間」にだけ使う。

### ポインター操作を使う

ホバーは、補助的な予告や反応に使う。

- クリック後の結果を軽く予告する
- 選択可能であることを示す
- どの要素へ焦点があるか示す
- 同じ反応をキーボードフォーカスでも提供する
- ホバーしないと読めない情報を置かない

### 余白を編集レーンとして使う

PCの余白には、章の位置、補足、進行方向を置ける。ただし、本文と競合する固定要素を増やさない。

余白の要素は次の条件を満たす。

- 本文を隠さない
- 読み順を変更しない
- 主要情報ではない
- 動きが止まっても役割が分かる

### stickyを使う条件

`position: sticky`は、同じ対象を見続けながら説明だけを更新する場合に使う。

次の場合は使わない。

- 単に見た目が豪華になる
- スクロール距離を増やすだけ
- 章ごとに対象が変わる
- 高さの短い画面で本文が詰まる
- 固定解除の位置が分かりにくい

## モバイルでの使い方

- 見出し→本文→具体例の順で視界へ入れる
- 比較は左右だけでなく、Before→Afterの時間差でも見せる
- 補足は帯へ詰めず、本文の後に遅れて置く
- ユーモアは固定表示せず、一度だけ通り過ぎる
- スクロールを奪わず、通常の読み進め方を守る
- 長い説明の途中に、内容を補強する小休止を置く
- 小休止の後には、本文へ戻る合図を置く

モバイル版をPC版の一列化として作らない。狭い画面は、失敗したPCではない。

## 速度とイージング

数値は固定テンプレートではなく、役割の基準として使う。

### productive motion

対象：

- ホバー
- フォーカス
- 開閉
- 選択
- コピー完了
- 小さな進捗表示

目安：

- 160〜280ms
- 速く立ち上がり、滑らかに止まる
- 繰り返し操作しても待たされない

### expressive motion

対象：

- 章の転換
- 比較の成立
- 視点の反転
- オチ
- 本文への帰還

目安：

- 420〜720ms
- 一つの領域で1〜2回まで
- バウンドや過剰なオーバーシュートを標準にしない

### designmdでの基準値

```css
:root {
  --motion-productive-fast: 160ms;
  --motion-productive: 240ms;
  --motion-expressive: 560ms;
  --motion-expressive-slow: 720ms;
  --motion-ease-standard: cubic-bezier(.2, 0, 0, 1);
  --motion-ease-enter: cubic-bezier(0, 0, 0, 1);
  --motion-ease-exit: cubic-bezier(.4, 0, 1, 1);
}
```

すべてを同じ`ease`と同じ時間で動かさない。

## 実装

推奨：

- `IntersectionObserver`で初回登場を検知する
- `transform`と`opacity`を中心に使う
- DOMの読み順を変更しない
- 自動再生は短く、一度で止める
- JavaScriptがなくても同じ情報を読める
- CSSとJavaScriptはHTMLから静的に参照し、並列で取得する
- 複数の拡張スクリプトがDOMを組み替える場合、最終状態まで初期表示を抑制する
- 初期表示の抑制には必ずタイムアウトによる解除を用意する
- アセットURLへ版を付け、旧キャッシュとの混在を避ける
- PCとモバイルで別のメディアクエリと演出責任を持つ
- JavaScriptで動きを追加する場合、同じ内容を静止状態でも確認できるようにする
- スクロール連動処理は`requestAnimationFrame`でまとめる
- 独自の動作軽減設定を保存する場合、保存失敗時もページを壊さない

避ける：

- スクロールを奪う演出
- 大きなパララックス
- 終わらない揺れ、点滅、回転
- 本文を読むたびに毎回発火する演出
- 高さや幅を動かしてレイアウトをずらす
- JavaScriptからCSSを一枚ずつ後付けする
- 素のHTML、途中版、完成版を順番に見せる
- PCとモバイルへ同じ移動距離、同じ速度を適用する
- すべての章を同じ方向から表示する
- ホバーだけで意味や操作を成立させる

## 初期表示とちらつき

動きの品質は、動き始める前から決まる。

HTMLを表示した後でCSSや複数世代のJavaScriptを順番に適用すると、読み手には次の状態が見える。

1. 素のHTML
2. 基礎CSSだけの状態
3. 拡張途中の状態
4. 最終状態

これはフォントのちらつきに見えても、実際にはDOMとスタイルの途中経過が見えている場合がある。

対策：

1. 最終状態で必要なCSSを`head`から静的に読み込む
2. 依存順を保った`defer`スクリプトとしてHTMLへ記述し、取得自体は並列化する
3. 最初の描画前に`is-booting`を付ける
4. 全拡張スクリプトの実行後、二回の`requestAnimationFrame`を経て解除する
5. 読み込み失敗に備え、数秒後に必ず解除する
6. `prefers-reduced-motion`の判定も最初の描画前に行う

```html
<style>
  html.is-booting body { opacity: 0; pointer-events: none; }
</style>
<script>
  document.documentElement.classList.add('is-booting');
  setTimeout(() => document.documentElement.classList.remove('is-booting'), 3000);
</script>
```

最終状態を一度だけ見せる。舞台裏を見せる企画でない限り、着替え途中は舞台裏でよい。

## アクセシビリティ

システム設定で動きを減らした利用者には、非本質的な移動を削除または置換する。

```css
@media (prefers-reduced-motion: reduce) {
  *, *::before, *::after {
    animation-duration: .01ms !important;
    animation-iteration-count: 1 !important;
    transition-duration: .01ms !important;
    scroll-behavior: auto !important;
  }
}
```

ページ内に独自の「動きを減らす」操作を設ける場合：

- OS設定を優先する
- ページ内設定を`aria-pressed`などで伝える
- 解除後に隠れた要素が残らないようにする
- 設定保存に失敗しても現在の表示は維持する
- 動きを戻しても、読了済みの演出を強制再生しない

## レビュー

- 動きを削除したとき、失われる役割を説明できるか
- 全セクションが同じフェードになっていないか
- productive motionとexpressive motionを分けているか
- 横移動の方向に意味があるか
- 複数要素が周辺視野で同時に動いていないか
- ユーモアが情報より前に出ていないか
- ためが単なる待ち時間になっていないか
- オチの後に本文へ戻っているか
- stickyが通常のスクロールより理解を増やしているか
- ホバーしなくても情報へ到達できるか
- キーボードフォーカスでも同じ反応があるか
- 固定要素が本文を隠していないか
- `prefers-reduced-motion: reduce`で内容を失わないか
- ページ独自の動作軽減設定が壊れていないか
- JavaScriptを無効にしても本文を読めるか
- 初回表示で素のHTMLや途中版が見えていないか
- 読み込み失敗時にページが非表示のまま残らないか
- `transform`と`opacity`以外の連続アニメーションが必要か
- PCとモバイルの演出が単なる拡大縮小になっていないか

## 安定化条件

このガイドはbetaである。次を満たした場合にstableへ移行する。

- PCとスマートフォンの双方で読み進める動機が増える
- 読解時間や操作を不必要に増やさない
- 動きを減らす設定で同じ情報へ到達できる
- 動きが新しいテンプレートとして全ページへ増殖しない
- 初期表示で完成前の状態が見えない
- productive / expressiveの区別が実案件で再現できる
- 横移動やstickyを使った理由を説明できる
