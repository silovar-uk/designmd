# ページモーション設計

> Status: beta  
> Scope: HTML版、長い読み物、教材、説明ページ

ページモーションは、要素を派手に動かすための装飾ではない。

> 読み手が、いつ気づき、いつ止まり、いつ笑い、いつ内容へ戻るかを、時間として編集する。

特にスマートフォンでは、PCのような左右の広がりを使いにくい。幅の代わりに、登場順、間、スクロールによる回収を使って情報の差をつくる。

## 原則

1. 動きには編集上の役割を持たせる
2. 主要情報を動きだけに依存させない
3. 一つの大きな領域に、強い動きは1〜2回まで
4. 同じフェードを全セクションへ配らない
5. 動いた後は止める
6. オチの後は静かな状態へ戻る
7. `prefers-reduced-motion`を尊重する
8. 初期表示では完成前のDOMや旧スタイルを見せない

全部を動かすと、全部が背景になる。かなり元気な背景である。

## 五つの動き

### 1. 登場

新しい章、対象、視点が現れたことを知らせる。見出しと本文をわずかにずらし、初回だけ実行する。

### 2. 引き継ぎ

前の要素から次の要素へ、関係や視点が移ったことを示す。線やラベルを動かす場合も、何を渡しているかを言葉で説明できることを条件にする。

### 3. ため

直後の発見、反論、オチの前に、一呼吸ぶんの時間差をつくる。重要条件や安全情報を演出のために遅らせない。

### 4. オチ

予想を一度だけ外し、理解や記憶を強める。読者を笑いの対象にせず、すべての見出しをオチにしない。

### 5. 帰還

動きやユーモアの後で、本文へ戻ったことを知らせる。笑いが終わったのに、ページだけが踊り続けない。

## モバイルでの使い方

- 見出し→本文→具体例の順で視界へ入れる
- 比較は左右だけでなく、Before→Afterの時間差でも見せる
- 補足は帯へ詰めず、本文の後に遅れて置く
- ユーモアは固定表示せず、一度だけ通り過ぎる
- スクロールを奪わず、通常の読み進め方を守る
- 長い説明の途中に、内容を補強する小休止を置く
- 小休止の後には、本文へ戻る合図を置く

モバイル版をPC版の一列化として作らない。狭い画面は、失敗したPCではない。

## オモコロから借りるもの

オモコロには、記事ごとに通常のサイト形式を曲げる例や、3Dなど固有の操作を組み込みながら、動作しない環境でも本文を読める例がある。

参考：

- 景色のカラーコード  
  https://omocoro.jp/kiji/557009/
- ダブル  
  https://omocoro.jp/kiji/557544/
- オモコロ特集  
  https://omocoro.jp/kiji/

借りる考え方：

- 記事・題材固有の仕掛けを許す
- 読者の予想を一度だけ外す
- 本文、画像、操作を一つの企画として扱う
- 仕掛けが動かなくても本文は読める
- 笑いを単独で置かず、その直前の内容を受けて成立させる

借りないもの：

- 固有のロゴ、書体、キャラクター、配色
- タイトルやギャグの調子
- 題材と無関係な奇抜さ
- すべてのページを特殊サイトにする発想

## 実装

推奨：

- `IntersectionObserver`で初回登場を検知する
- `transform`と`opacity`を中心に使う
- DOMの読み順を変更しない
- 自動再生は短く、一度で止める
- JavaScriptがなくても同じ情報を読める
- CSSとJavaScriptはHTMLから静的に参照し、並列で取得する
- 複数の拡張スクリプトがDOMを組み替える場合、最終状態まで初期表示を抑制する
- 初期表示の抑制には必ずタイムアウトによる解除を用意する
- アセットURLへ版を付け、旧キャッシュとの混在を避ける

避ける：

- スクロールを奪う演出
- 大きなパララックス
- 終わらない揺れ、点滅、回転
- 本文を読むたびに毎回発火する演出
- 高さや幅を動かしてレイアウトをずらす
- JavaScriptからCSSを一枚ずつ後付けする
- 素のHTML、途中版、完成版を順番に見せる

## 初期表示とちらつき

動きの品質は、動き始める前から決まる。

HTMLを表示した後でCSSや複数世代のJavaScriptを順番に適用すると、読み手には次の状態が見える。

1. 素のHTML
2. 基礎CSSだけの状態
3. 拡張途中の状態
4. 最終状態

これはフォントのちらつきに見えても、実際にはDOMとスタイルの途中経過が見えている場合がある。

対策：

1. 最終状態で必要なCSSを`head`から静的に読み込む
2. 依存順を保った`defer`スクリプトとしてHTMLへ記述し、取得自体は並列化する
3. 最初の描画前に`is-booting`を付ける
4. 全拡張スクリプトの実行後、二回の`requestAnimationFrame`を経て解除する
5. 読み込み失敗に備え、数秒後に必ず解除する
6. `prefers-reduced-motion`の判定も最初の描画前に行う

```html
<style>
  html.is-booting body { opacity: 0; pointer-events: none; }
</style>
<script>
  document.documentElement.classList.add('is-booting');
  setTimeout(() => document.documentElement.classList.remove('is-booting'), 3000);
</script>
```

最終状態を一度だけ見せる。舞台裏を見せる企画でない限り、着替え途中は舞台裏でよい。

## アクセシビリティ

システム設定で動きを減らした利用者には、非本質的な移動を削除または置換する。

```css
@media (prefers-reduced-motion: reduce) {
  *, *::before, *::after {
    animation-duration: .01ms !important;
    animation-iteration-count: 1 !important;
    transition-duration: .01ms !important;
    scroll-behavior: auto !important;
  }
}
```

参考：

- Media Queries Level 5 / prefers-reduced-motion  
  https://www.w3.org/TR/mediaqueries-5/#prefers-reduced-motion
- WCAG 2.2 / Animation from Interactions  
  https://www.w3.org/WAI/WCAG22/Understanding/animation-from-interactions
- WCAG Technique C39  
  https://www.w3.org/WAI/WCAG22/Techniques/css/C39

## レビュー

- 動きを削除したとき、失われる役割を説明できるか
- 全セクションが同じフェードになっていないか
- ユーモアが情報より前に出ていないか
- ためが単なる待ち時間になっていないか
- オチの後に本文へ戻っているか
- 固定要素が本文を隠していないか
- `prefers-reduced-motion: reduce`で内容を失わないか
- JavaScriptを無効にしても本文を読めるか
- 初回表示で素のHTMLや途中版が見えていないか
- 読み込み失敗時にページが非表示のまま残らないか

## 安定化条件

このガイドはbetaである。次を満たした場合にstableへ移行する。

- スマートフォンで読み進める動機が増える
- 読解時間や操作を不必要に増やさない
- 動きを減らす設定で同じ情報へ到達できる
- 動きが新しいテンプレートとして全ページへ増殖しない
- 初期表示で完成前の状態が見えない
