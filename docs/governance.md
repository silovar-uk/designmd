# バージョン・状態・更新運用

> 状態：stable  
> designmd版：2026.07.16-beta  
> 最終更新：2026-07-16

## 1. バージョン

beta期間は日付ベースのバージョンを使う。

```text
YYYY.MM.DD-beta
```

大きな互換性変更を伴う正式版へ移行する時点で、セマンティックバージョニングへの変更を検討する。

## 2. 状態

| 状態 | 意味 | 利用方法 |
|---|---|---|
| stable | 通常案件で使う基準 | 原則として適用する |
| beta | 方向は確定しているが改善中 | 適用し、問題を記録する |
| experimental | 効果が未検証 | 試行案件を限定し、撤回可能にする |
| deprecated | 新規利用を推奨しない | 代替先へ移行する |

`experimental` を、目新しい機能という意味で使わない。評価方法と撤回条件が書かれていることを条件とする。

## 3. 正典と派生物

- `docs/` の正典：概念の詳細
- `DESIGN.md`：外部AIへ渡す自己完結ダイジェスト
- `README.md`：入口と索引
- `site/`：人が操作しながら理解する派生表示
- `prompts/`：実行用テンプレート

正典を変更したら、DESIGN.md、入口、関連プロンプト、サイト表示の同期要否を確認する。

## 4. 変更履歴

- 正典の追加・削除・意味変更は `CHANGELOG.md` へ記載する
- 誤字やリンク修正は、意味が変わらない限り省略できる
- experimentalの追加には、評価期限と撤回条件を書く
- stableへの昇格には、DESIGN.md、README、プロンプト、site、VERSIONの同期を確認する
- deprecated化には、代替先と移行期限を書く

## 5. Markdownリリース

`docs-v*` タグを作ると、GitHub Actionsが次をまとめたZIPを作り、GitHub Releaseへ添付する。

- DESIGN.md
- README.md
- CHANGELOG.md
- VERSION
- docs/
- prompts/

手動実行では、同じZIPをActionsのartifactとして確認できる。自動で毎週公開せず、意味のまとまった変更単位でリリースする。

## 6. リリース前チェック

- VERSIONとCHANGELOGが更新されている
- 正典所有表とリンクが一致する
- experimentalの評価期限がある
- DESIGN.mdへ要約が同期されている
- stableへ昇格した機能がREADME、プロンプト、siteへ反映されている
- サイトがJavaScriptなしでも基礎内容を読める
- 出典・加工表示が必要な参照に記載がある

## 7. 責任境界

ガイドの状態がstableでも、成果物の正確性、法令・権利、アクセシビリティ、公開可否を保証するものではない。stableは「ガイド内部で通常利用する状態」を示し、外部規格への適合認証ではない。
