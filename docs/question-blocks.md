# 逆質問は、仮説付きテキストブロックで返す

制作前に逆質問が必要な場合、質問は通常の箇条書きで散らして返さず、ユーザーがそのまま回答を書き込める **テキストブロック** として提示します。

ただし、空欄だけを並べて、ユーザーへゼロから考えさせないでください。

> 逆質問は、情報を取りに行くだけのアンケートではない。制作者側の仮説を先に見せ、方向を一緒に選ぶためのGATE 0である。

目的は次の三つです。

- 回答負荷を下げる
- 制作者側の理解と仮説を早い段階で検証する
- 回答を、そのまま制作ログやプロンプトへ再利用できる形にする

---

## 1. 最重要ルール

### 1.1 ゼロから埋めさせない

悪い逆質問は、空欄だけを渡します。

```text
1. 読み手：
   回答：

2. 雰囲気：
   回答：
```

これでは、依頼者が制作ブリーフまで自分で設計することになります。

良い逆質問は、会話から読み取った仮説、候補、推奨案を先に置きます。

```text
1. 主な読み手：
   こちらの仮説：AIを使って資料を作る個人・企画担当者
   他の候補：経営層／デザイナー／一般のAIユーザー
   推奨：一般のAIユーザーまで広げる
   理由：専門家向けの方法論ではなく、真似できる事例として見せたい依頼だから
   回答：推奨で進める／修正：
```

ユーザーは、合っている部分を再入力せず、違うところだけ直せます。

### 1.2 先に仮説を出す

逆質問の前に、会話から次を読み取ります。

- すでに確定している事実
- 明示された好み
- 過去のフィードバック
- 今回の成果物から自然に推測できる条件
- まだ確定していないが、有力な方向

確定事項と仮説は分けて書きます。

```text
【すでに確定している前提】
- 成果物：PPTXとPDF
- 読後の状態：真似したいと思ってもらう

【こちらの仮説】
- 発表だけでなく、PDF単体でも読む可能性がある
- 事例紹介なので、方法論より失敗と変化を中心にした方が伝わる
```

仮説を事実のように断定しないでください。

### 1.3 具体案を2〜4個出す

方向性を聞く質問では、原則として2〜4案を提示します。

案は、今回の題材に合わせて具体化します。

悪い例：

```text
雰囲気：明るい／暗い／普通
```

良い例：

```text
1. デザインの主方向：
   A｜編集プロダクト
      白・黒・朱色。大きな日本語、実物、赤入れ、雑誌的な余白。
   B｜プロダクトローンチ
      黒・白中心。巨大な一文、画面キャプチャ、速度のある展開。
   C｜カルチャー誌
      切り抜き、欄外コメント、少し奔放。ユーモアを強める。
   推奨：Aを70％＋Bを30％
   理由：事例としての信頼感と、ベンチャー的な推進力を両立できるため
   回答：推奨で進める／修正：
```

### 1.4 推奨案と理由を必ず置く

選択肢だけを並べて判断を丸投げしません。

各質問では、可能な限り次を示します。

- 推奨案
- 推奨する理由
- 採用した場合に得られるもの
- 必要なら、失うものやリスク

```text
3. 説明量：
   A｜12〜14枚。口頭説明を中心にする。
   B｜16〜18枚。PDF単体でも理解できる。
   C｜14〜16枚。基本は軽快にし、事例ページだけ厚くする。
   推奨：C
   理由：わくわく感を保ちながら、事例紹介として必要な説明量を確保できるため
   回答：推奨で進める／修正：
```

### 1.5 「違うところだけ直せる」回答欄にする

回答欄は、単なる `回答：` より、次の形を優先します。

```text
回答：推奨で進める／修正：
```

複数項目をまとめて提示する場合は、ブロック冒頭に次を添えます。

```text
空欄は推奨案で進めます。違うところだけ書き換えてください。
```

ユーザーがすべて書き直す必要をなくします。

### 1.6 選択肢で誘導しすぎない

提案は、ユーザーの思考を助けるためのものです。選択肢の中へ閉じ込めるためではありません。

次を守ります。

- `その他` や `組み合わせ` を許可する
- 推奨案を唯一の正解として扱わない
- ユーザーの言葉が出たら、既存の選択肢より優先する
- 提案へ都合よく誘導するために、弱い案を並べない

### 1.7 事実は無理に推測しない

日時、人数、予算、公開範囲、権利条件など、誤ると危険な事実は空欄で確認して構いません。

その場合も、分かっている範囲は先に記入します。

```text
4. 発表時間：
   こちらで分かっていること：PPTXとPDFの両方を制作する
   確認したいこと：実際の登壇時間
   回答：
```

推測できる好みと、確認が必要な事実を混同しないでください。

## 2. 出力形式

### 2.1 テキストブロックで提示する

逆質問は、原則として `text` コードブロックで返します。

```text
【すでに確定している前提】
- テーマ：
- 成果物：
- 既に分かっている要望：

【こちらの仮説】
- 

空欄は推奨案で進めます。違うところだけ書き換えてください。

【必須】
1. 主な読み手：
   こちらの仮説：
   他の候補：
   推奨：
   理由：
   回答：推奨で進める／修正：

2. 使用場面：
   A｜
   B｜
   C｜
   推奨：
   理由：
   回答：推奨で進める／修正：

3. 読後に求める理解・判断・行動：
   こちらの仮説：
   推奨：
   理由：
   回答：推奨で進める／修正：

【推奨】
4. トーン・温度：
   A｜
   B｜
   C｜
   推奨：
   理由：
   回答：推奨で進める／修正：

【任意】
5. 絶対に避けたい表現：
   こちらの仮説：
   回答：その通り／追加：
```

### 2.2 分類は三つまで

質問は、次の三分類にします。

- **必須**：回答がないと方向性がずれる
- **推奨**：回答があると品質が上がる
- **任意**：こだわりがあれば反映する

分類を増やしすぎないでください。

### 2.3 質問数は最大8問

質問は最大8問に絞ります。

必要な情報が多い場合でも、まず方向性を決める質問を優先します。

一つの質問へ、関係の近い選択肢をまとめることで、質問数を減らします。

### 2.4 既に分かっていることは聞かない

ユーザーがすでに明示した情報は、質問に戻さず、前提として記入します。

悪い例：

```text
ユーザー：PPTとPDFで作って。
AI：成果物の形式は何ですか。
```

良い例：

```text
【すでに確定している前提】
- 成果物：PPTXとPDF
```

## 3. 質問の種類ごとの作り方

### 3.1 読み手

対象者名だけでなく、今回の内容に対する距離を提案します。

```text
1. 主な読み手：
   A｜AIを日常的に使う制作者
   B｜AIは使うが、資料制作の方法論までは持たない人
   C｜AI活用事例を探している一般ユーザー
   推奨：B＋C
   理由：「高度な設計論」より「真似できる事例」として見せたいから
   回答：推奨で進める／修正：
```

### 3.2 使用場面

媒体名だけでなく、読み方の違いを示します。

```text
2. 使用場面：
   A｜登壇中心。画面は軽く、台本で深く説明する。
   B｜PDF共有中心。資料単体で理解できる情報を残す。
   C｜登壇＋後日共有。基本は軽快にし、要点と事例は資料内にも残す。
   推奨：C
   理由：PPTXとPDFの両方が成果物に含まれるため
   回答：推奨で進める／修正：
```

### 3.3 トーン・デザイン

抽象形容詞ではなく、画面の挙動まで説明します。

```text
3. トーン：
   A｜編集スタジオ
      大きな日本語、原稿、赤入れ、実物を中心にする。
   B｜プロダクトローンチ
      宣言、速度、画面キャプチャ、強い転換を中心にする。
   C｜研究発表
      仮説、実験、失敗、更新履歴を中心にする。
   推奨：Aを中心に、Bの速度とCの検証感を加える
   回答：推奨で進める／修正：
```

### 3.4 ユーモア

強さだけでなく、笑いの対象と方法を示します。

```text
4. ユーモア：
   A｜欄外の小さなツッコミ
   B｜自分たちの失敗を見せる自虐
   C｜一度だけページ構成そのものを崩す
   推奨：A＋B。Cは転換点の1ページだけ。
   理由：軽さは出しつつ、事例の信頼性を壊さないため
   回答：推奨で進める／修正：
```

## 4. 悪い出し方

```text
【必須】
1. 読み手：
   回答：

2. 使用場面：
   回答：

3. 雰囲気：
   回答：

4. 枚数：
   回答：
```

問題：

- 依頼者がゼロから制作ブリーフを作る必要がある
- 制作者側が依頼をどう理解したか分からない
- 選択肢がなく、抽象語で答えるしかない
- 推奨案がなく、判断を丸投げしている
- 既知情報と不足情報が混ざりやすい

---

## 6. 制作フローとの関係

逆質問テキストブロックは、GATE 0の成果物です。

回答後は、次へ進みます。

1. 回答と採用された推奨案を前提として要約する
2. 修正された箇所を制作条件へ反映する
3. 足りない原材料を確認する
4. 構成案を最低3案出す
5. 推奨案を示す
6. ユーザーが構成を選ぶ
7. 代表ページへ進む

質問ブロックを出した時点で、完成版の制作へ進まないでください。

---

## 7. 例外

ユーザーが明確に「質問せず進めて」「一括で」「仮定して作って」と指定した場合は、質問ブロックを省略して構いません。

ただし、その場合も提出時に、置いた前提と採用した仮説をテキストブロックで示します。

```text
【置いた前提】
- 読み手：
- 使用場面：
- 求める理解・判断・行動：
- 形式：
- トーン：
- 採用した推奨案：
- 推奨理由：
- 未確認事項：
```
