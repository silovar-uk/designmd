# 逆質問はテキストブロックで返す

制作前に逆質問が必要な場合、質問は通常の箇条書きで散らして返さず、ユーザーがそのまま回答を書き込める **テキストブロック** として提示します。

目的は、質問へ答えやすくすることと、回答をそのまま制作ログやプロンプトへ再利用できるようにすることです。

> 逆質問は、会話の途中に置くアンケートではない。制作のGATE 0で使う入力フォームである。

---

## 1. 基本ルール

### 1.1 質問はコードブロックで提示する

逆質問は、原則として次の形式で返します。

```text
【必須】
1. 読み手：
   回答：

2. 使用場面：
   回答：

3. 読後に求める理解・判断・行動：
   回答：

【推奨】
4. 避けたい表現・雰囲気：
   回答：

5. 残したい原材料：
   回答：

【任意】
6. こだわり：
   回答：
```

### 1.2 分類は三つまで

質問は、次の三分類にします。

- **必須**：回答がないと方向性がずれる
- **推奨**：回答があると品質が上がる
- **任意**：こだわりがあれば反映する

分類を増やしすぎないでください。

### 1.3 質問数は最大8問

質問は最大8問に絞ります。

必要な情報が多い場合でも、まず方向性を決めるための質問を優先します。

### 1.4 回答欄を付ける

各質問には、`回答：` を付けます。

ユーザーが引用返信しなくても、そのまま埋められる形にします。

### 1.5 選択肢がある場合は併記する

回答しやすくするため、必要に応じて選択肢を入れます。

```text
4. 使用場面：
   選択肢：発表用／閲覧用／会議用／営業用／学習用／Web公開用
   回答：
```

### 1.6 既に分かっていることは聞かない

ユーザーがすでに明示した情報は、質問に戻さず、前提として記入します。

```text
【こちらで置いた前提】
- テーマ：designmdの思想紹介
- 成果物：PPTとPDF
- 重要課題：ユーモア、わくわく感、ベンチャー感の不足
```

そのうえで、足りない点だけを聞きます。

---

## 2. 逆質問の出力テンプレート

```text
【こちらで置いた前提】
- テーマ：
- 成果物：
- 既に分かっている要望：

【必須】
1. 読み手は誰ですか。
   例：社内の上司／同僚／AIを使う制作者／一般公開／採用候補者
   回答：

2. どの場で使いますか。
   例：登壇して説明／PDF共有／Web公開／社内提案／ワークショップ
   回答：

3. 見た人に、最後どうなってほしいですか。
   例：共感する／使いたくなる／導入を認める／開発に参加したくなる
   回答：

【推奨】
4. 今回いちばん出したい温度はどれですか。
   例：創業者の原体験／プロダクトローンチ／実験発表／チームのキックオフ／投資家向けの未来像
   回答：

5. ユーモアの強さはどこまで許容しますか。
   例：静かな微笑み／軽いツッコミ／自虐を含む笑い／大胆な実験
   回答：

6. 実例や失敗をどこまで見せてよいですか。
   例：実物を見せる／再現図にする／文章だけで触れる／出さない
   回答：

【任意】
7. 参考にしたい媒体や雰囲気はありますか。
   例：Speaker Deck／ゆる言語学ラジオ／オモコロ／スタートアップのローンチ資料
   回答：

8. 絶対に避けたい表現はありますか。
   例：黒背景の意識高い資料／巨大英字／過度な自虐／整いすぎたカードUI
   回答：
```

---

## 3. 悪い出し方

```text
質問です。
- 誰に見せますか？
- どんな雰囲気がいいですか？
- 何枚くらいですか？
```

問題：

- 回答欄がない
- どれが必須か分からない
- そのままコピーして返しにくい
- 既知情報と不足情報が混ざりやすい

---

## 4. 良い出し方

```text
【こちらで置いた前提】
- 成果物：PPTとPDF
- 目的：designmdの思想を説明する
- 現時点の課題：ユーモア、わくわく感、ベンチャー感が不足

【必須】
1. 主な読み手：
   回答：

2. 使用場面：
   回答：

3. 見た人に起こしたい変化：
   回答：

【推奨】
4. ベンチャー感の方向：
   選択肢：創業者の原体験／プロダクトローンチ／実験発表／チームのキックオフ／未来像
   回答：

5. ユーモアの強さ：
   選択肢：静かな微笑み／軽いツッコミ／明確な笑い／大胆な実験
   回答：
```

---

## 5. 制作フローとの関係

逆質問テキストブロックは、GATE 0の成果物です。

回答後は、次へ進みます。

1. 回答を前提として要約する
2. 足りない原材料を確認する
3. 構成案を最低3案出す
4. 推奨案を示す
5. ユーザーが構成を選ぶ
6. 代表ページへ進む

質問ブロックを出した時点で、完成版の制作へ進まないでください。

---

## 6. 例外

ユーザーが明確に「質問せず進めて」「一括で」「仮定して作って」と指定した場合は、質問ブロックを省略して構いません。

ただし、その場合も提出時に、置いた前提をテキストブロックで示します。

```text
【置いた前提】
- 読み手：
- 使用場面：
- 求める理解・判断・行動：
- 形式：
- トーン：
- 未確認事項：
```
