# 制作は、最初の一問から始める

> 状態：stable  
> designmd版：2026.07.19-beta  
> 最終更新：2026-07-19

DesignMDを使う制作では、計画、調査、構成、制作へ入る前に、必ず一問だけ質問する。

> 最初の一問は、不足情報を全部集めるためではない。成果物を最も大きく変える判断を、依頼者と制作者が最初に共有するためにある。

質問は一問だけで終える。回答後は追加質問を重ねず、残る不足は仮説、未確認事項、制作範囲の境界として扱う。

---

## 1. 最重要ルール

### 1.1 最初の応答で、一問だけ聞く

最初の応答では、計画、長い分析、制作案を先に出さない。

```text
最初に一つだけ確認します。
今回、最優先にするのは A｜理解、B｜感情、C｜行動 のどれですか？ 推奨はBです。
```

質問文は一つだけにする。

- 疑問文を二つ並べない
- 「誰に、どこで、何を、どのくらい」のような複合質問にしない
- 質問符を複数使わない
- 箇条書きの各項目を質問にしない
- 回答後に追加質問をしない

### 1.2 既に分かっていることは聞かない

会話、添付資料、過去のフィードバック、明示された条件を先に読む。

既知情報を聞き直すことは、一問を浪費する行為として扱う。

悪い例：

```text
ユーザー：PPTXとPDFで作ってください。
AI：成果物の形式は何ですか？
```

良い例：

```text
最初に一つだけ確認します。
PPTXとPDFの両方で使う前提で、発表時の勢いとPDF単体の読みやすさのどちらを優先しますか？ 推奨はPDF単体でも読める設計です。
```

### 1.3 最も結果を変える未知を選ぶ

一問は、答えによって構成、表現、調査範囲、WAO、納品方法のいずれかが大きく変わるものを選ぶ。

優先順位は次の通り。

1. 安全、法令、権利、公開可否に関わる必須事実
2. 読み手と使用場面
3. 読後に変えたい理解、感情、判断、行動
4. 採用する問い、中心メッセージ
5. WAOで上振れさせたい期待
6. 編集型、資料モード、表現の強度
7. 色、書体、細かな好み

下位の好みより、上位の目的を優先する。

### 1.4 仮説と推奨を添える

質問だけを丸投げしない。

- 会話から読み取った仮説
- 2〜3個の具体的な選択肢
- 推奨案
- 短い推奨理由

を質問の中へ圧縮する。

```text
最初に一つだけ確認します。
今回の主な読者は A｜実務担当者、B｜意思決定者、C｜一般の利用者 のどれに置きますか？ 内容の具体性を生かせるため、推奨はAです。
```

選択肢は回答負荷を下げるために使い、選択肢の外の回答も許容する。

### 1.5 回答しやすくする

回答は、記号、一語、短い修正だけで成立させる。

```text
回答例：B
回答例：推奨で
回答例：A寄り。ただし経営層にも伝わるように
```

初回質問のために長いテキストブロックやアンケートを出さない。

## 2. 一問を選ぶ手順

### STEP 1：確定事項を内部で整理する

質問として表示せず、内部で次を整理する。

- 依頼された成果物
- 読み手
- 使用場面
- 読後の変化
- 固定条件
- 過去の好みと修正履歴
- 公開範囲
- WAO、編集型、質感に関する希望

### STEP 2：未確定事項を列挙する

候補となる不足を内部で複数出す。

### STEP 3：影響度で一つへ絞る

次の三基準で選ぶ。

- 回答が変わると、構成が変わるか
- 回答が変わると、制作物の役割が変わるか
- 間違って仮定した場合、やり直しや誤解が大きいか

三つとも弱い質問は聞かない。

### STEP 4：具体案と推奨を付ける

抽象語だけを並べない。今回の題材に合わせて具体化する。

### STEP 5：一問だけ送る

質問の回答を受ける前に、計画、調査結果、完成案へ進まない。

## 3. 質問の形式

### 基本形

```text
最初に一つだけ確認します。
今回、［成果物を最も変える選択］は A｜［具体案］、B｜［具体案］、C｜［具体案］のどれにしますか？ 推奨は［案］です。理由は［短い理由］です。
```

### 読み手を確認する

```text
最初に一つだけ確認します。
主な読み手は A｜現場の実務担当者、B｜社内の意思決定者、C｜一般の利用者 のどれに置きますか？ 実例の細部を判断へつなげやすいため、推奨はAです。
```

### 使用場面を確認する

```text
最初に一つだけ確認します。
今回は A｜登壇時の勢い、B｜PDF単体の理解、C｜両方の中間 のどれを優先しますか？ PPTXとPDFを併用するため、推奨はCです。
```

### 読後の変化を確認する

```text
最初に一つだけ確認します。
読後に最も変えたいのは A｜理解、B｜判断、C｜行動 のどれですか？ 今回は実装へつなげる依頼なので、推奨はCです。
```

### WAOを確認する

```text
最初に一つだけ確認します。
WAOで最優先に上振れさせたいのは A｜理解、B｜感情、C｜行動、D｜記憶 のどれですか？ 今回は学びとして残す目的が強いため、推奨は理解と記憶です。
```

選択肢が四つ必要な場合も、疑問文は一つにする。

### 安全・権利条件を確認する

```text
最初に一つだけ確認します。
この素材は A｜公開利用可能、B｜社内限定、C｜権利未確認 のどれですか？ 未確認の場合は、公開版へ使用せず代替素材で進めます。
```

## 4. 回答後の進め方

回答を受けたら、以後は追加質問をしない。

1. 回答を制作条件へ反映する
2. 既知情報と置いた仮説を整理する
3. 計画または構成案を提示する
4. 不足情報は合理的な仮定で補う
5. 仮定した箇所を制作ログへ残す
6. 危険な未確認事項は、制作範囲から外すか要確認として明示する
7. 代表ページ、全体制作、レビューへ進む

回答後に別の不足へ気づいても、質問を追加しない。

```text
【置いた前提】
- 

【未確認事項】
- 

【今回の制作範囲から外すもの】
- 
```

として扱う。

## 5. WAOとの関係

最初の一問そのものにも、無理のないWAOを持たせる。

- 依頼を理解していることが伝わる
- 選択肢が題材固有である
- 推奨があり、答えやすい
- 一問に答えるだけで制作の見通しが良くなる

ただし、質問文をキャッチコピー化しない。驚かせることより、判断を前へ進めることを優先する。

---

## 6. 失敗パターン

- 最初に計画を長く説明してから質問する
- 一文の中へ複数の確認事項を詰め込む
- 一問と言いながら、AとBそれぞれについて理由を聞く
- 既に明示された成果物や読み手を聞き直す
- 色や雰囲気を、目的より先に聞く
- 選択肢だけを並べ、推奨を示さない
- 回答後に「もう一点だけ」と質問を増やす
- ユーザーが質問不要と言ったため、一問を省略する
- 安全や権利の必須条件を推測で埋める
- 質問の回答がなくても、勝手に完成版へ進む
