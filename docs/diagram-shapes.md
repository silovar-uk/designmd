# コンサル資料の図形・矢印・台本設計

対象：コンサルティング資料、企画書、業務フロー、戦略資料、会議資料、提案書、発表用PowerPoint。

このガイドは、図形や矢印をきれいに装飾するためのものではありません。

> 図形を名詞、矢印を動詞、囲みを文脈として使い、複雑な関係を短時間で読める状態にする。

また、最終的にスライドは人が説明するものです。図を作る前に、各ページで実際に話す内容を文章として書きます。

> スライドは台本の代用品ではない。台本を視覚的に圧縮し、話す順序と視線を支える道具である。

---

## 1. なぜ矢印が不自然に見えるのか

### 1.1 関係が決まる前に矢印を置いている

次の関係は、すべて別物です。

- 時間的に次へ進む
- 原因が結果を生む
- 情報を渡す
- 作業を引き継ぐ
- 前提として必要になる
- 影響を与える
- 部分が全体を構成する
- 選択肢から一つを選ぶ

これらを同じ右向き矢印で表すと、見た目は整っていても意味が曖昧になります。

### 1.2 箱を並べた後で、空いた場所を矢印で埋めている

図の意味は、箱そのものより箱同士の関係にあります。

箱を先に均等配置し、最後に矢印を足すと、次が起きます。

- 本来は並列なのに流れに見える
- 因果ではないのに因果に見える
- 全部が一本道に見える
- 矢印の始点と終点が曖昧
- 線が文字や別の図形を横切る

### 1.3 一つの図で矢印の意味が変わる

同じ太さ、同じ色、同じ形の矢印が、ある場所では時間、別の場所では因果、別の場所では移管を表している。

読み手は、矢印を見るたびに意味を推測し直す必要があります。

### 1.4 装飾として矢印を使う

- タイトルから本文へ向ける
- 強調したい数字を指す
- 余白を埋める
- ダイナミックに見せる
- 「つながっている感じ」を作る

この用途では、矢印より位置、余白、整列、色、注釈の方が適切な場合が多いです。

### 1.5 台本と図が別々に作られている

話す内容では「AがBを可能にする」と説明するのに、図ではAとBがただ横並びになっている。

逆に、図ではAからBへ太い矢印があるのに、説明ではその関係に触れない。

この不一致が、図をぎこちなく見せます。

## 2. 図の文法

### 図形は名詞

図形は、何が存在するかを表します。

- 人
- 組織
- 工程
- 状態
- 顧客
- 商品
- データ
- 判断
- 成果物

### 矢印は動詞

矢印は、図形同士に何が起きるかを表します。

- 生む
- 渡す
- 変える
- 必要とする
- 制約する
- 促進する
- 引き継ぐ
- 選ぶ
- 戻す

### 囲み・帯・レーンは文脈

- 同じフェーズ
- 同じ担当
- 同じ対象範囲
- 同じカテゴリ
- 現状／将来
- 社内／社外

### 色・太さ・大きさは修飾語

- 重要
- 未確認
- 変更対象
- 優先
- リスク
- 現在地

一つの視覚変数へ複数の意味を持たせないでください。

## 3. 描く前の「図の一文」

矢印を置く前に、関係を文章で書きます。

```text
主体Aが、条件Cのもとで、作用Vによって、対象Bを変える。
```

例：

> 問い合わせ内容を営業から運営へ引き継ぐ。

この場合：

- 図形A：営業
- 矢印V：引き継ぐ
- 図形B：運営
- ラベル：問い合わせ内容

一文にできない場合、関係が未定義です。

### 矢印文テスト

各矢印について、次を一文で言えるか確認します。

1. 何から
2. 何へ
3. 何が
4. どうなる
5. 条件は何か

言えない矢印は削除または再設計します。

---

## 4. 関係別の表現

| 表したい関係 | 第一候補 | 矢印の扱い |
|---|---|---|
| 時系列・工程 | 左から右、または上から下のプロセス | 方向を一つに統一 |
| 因果 | 原因→中間過程→結果 | 必要なら動詞を付ける |
| 入力・変換・出力 | 入力→処理→出力 | 処理を箱、変換を矢印または配置で示す |
| 情報・物の受け渡し | 担当別レーン＋接続線 | 何を渡すかラベルを置く |
| 依存関係 | 線または細い矢印 | 「必要」「前提」などを明示 |
| 階層・分解 | ツリー、枝線、インデント | 通常は矢印頭を付けない |
| 比較 | 横並び、表、マトリクス | 原則として矢印不要 |
| グルーピング | 近接、囲み、共通帯 | 矢印不要 |
| 選択・分岐 | 判断点＋分岐 | 分岐条件を必ず表示 |
| 合流 | 複数経路→合流点 | 合流条件を示す |
| フィードバック | 循環図 | 実際に戻る仕組みがある場合だけ使用 |
| 優先順位 | 位置、番号、大きさ | 矢印で順位を表さない |
| 現状から将来 | Before／After、移行ロードマップ | 大きな移行矢印は一つまで |

## 5. 矢印を使わない判断

以下では、矢印を使わない方が自然な場合があります。

### 並列

三つの要素が同じ階層なら、等間隔で並べるだけで十分です。

### 比較

現状と将来、A案とB案は、同じ軸の二列比較にします。

### 所属

要素が同じカテゴリに属するなら、囲み、背景帯、近接を使います。

### 強調

数字や結論へ注意を向ける場合、色、太字、余白、注釈線を使います。

### 階層

上位概念から下位概念への分解は、枝線やインデントで表せます。因果や時間と誤解されるため、むやみに矢印頭を付けません。

### 矢印削除テスト

矢印を消しても読み手が同じ関係を理解できるなら、削除を候補にします。

## 6. 矢印の基本ルール

### 6.1 一つの図では主方向を一つにする

- 工程：左→右
- 階層：上→下、または左→右
- 時間：左→右

途中で右→左、下→上へ折り返さないでください。

### 6.2 始点と終点を図形へ接続する

PowerPointのコネクタを使い、図形を動かしても線が追従する状態にします。

浮いた矢印、箱の手前で止まる矢印、箱の中心を突き抜ける矢印を避けます。

### 6.3 曲がりは少なくする

基本は直線または直角コネクタ。

一つの接続で何度も折り曲げる必要がある場合、配置か図の構造を見直します。

### 6.4 線を交差させない

交差は、関係の読み違いを生みます。

対策：

- ノードの順番を変える
- レーンを分ける
- 合流点を設ける
- 図を二枚に分ける
- 長い接続をラベル付きコネクタへ置換する

### 6.5 同じ関係は同じ線にする

- 実線：主要な関係
- 破線：補助、未確定、間接
- 強調色：今回説明する経路

など、意味を先に定義します。

### 6.6 矢印頭を大きくしすぎない

大きな矢印頭は、内容より線を目立たせます。

矢印は主役ではなく、関係を読む補助です。

### 6.7 太いブロック矢印を乱用しない

ブロック矢印が向く場面：

- 現状から将来への大きな移行
- 一つの明確な変換
- フェーズ全体の方向

細かな箱同士の接続には、通常の線コネクタを使います。

### 6.8 意味が自明でなければ動詞を付ける

例：

- Enables／可能にする
- Requires／前提とする
- Transfers／引き継ぐ
- Constrains／制約する
- Informs／判断材料になる

日本語の読み手には、必要がなければ日本語で書きます。

### 6.9 分岐には条件を書く

Yes／No、承認／差し戻し、基準以上／基準未満など。

無記名の二股矢印を作らないでください。

### 6.10 循環矢印は本当の循環だけ

循環図は「良いことが続く雰囲気」を表す装飾ではありません。

戻り先、戻る情報、更新の条件を説明できる場合だけ使用します。

## 7. 図形の意味

図形の種類を増やすほど、読み手が覚える文法も増えます。

基本は少数に絞ります。

| 図形 | 主な役割 | 注意 |
|---|---|---|
| 長方形 | 工程、主体、状態、要素 | 最も標準的 |
| 角丸長方形 | フェーズ、入口、出口、柔らかいまとまり | 長方形との意味差を決める |
| 円 | 起点、終点、節目、中心概念 | 装飾アイコンの背景にしない |
| ひし形 | 判断・分岐 | 「重要」や「強調」に使わない |
| 帯・背景面 | フェーズ、領域、担当 | 囲みすぎない |
| レーン | 担当、部署、システム | 引き継ぎの可視化に使う |
| 括弧・ブレース | 複数要素のまとめ、注釈 | 因果を表さない |
| シェブロン | 連続するフェーズ | 単なる3項目に使わない |
| 漏斗 | 絞り込み、選別、減少 | 量が減る理由を示す |
| ピラミッド | 階層、優先、蓄積 | 「なんとなく戦略的」で使わない |
| マトリクス | 二つの独立軸 | 軸名と方向を明示 |
| 循環 | 反復、フィードバック | 戻る仕組みを説明する |

### 同形同義

同じ形は、同じ役割に使います。

同じ長方形が「部署」「課題」「成果」を混在して表す場合は、見出し、色、レーンなどで役割を明確にします。

## 8. コンサル資料でよく使う図

### 8.1 プロセス

用途：工程、顧客体験、業務手順。

構成：

- 一方向
- 各箱は同じ粒度の行動
- 動詞＋目的語
- 分岐がある場合だけ判断点

悪い例：抽象名詞だけの箱を矢印でつなぐ。

良い例：

- 問い合わせを受ける
- 内容を分類する
- 担当部署へ引き継ぐ
- 回答を記録する

### 8.2 イシューツリー・ロジックツリー

用途：問いの分解、原因、解決策。

構成：

- 右へ進むほど詳細
- 各階層の問いを統一
- 枝は分解関係
- 原則として枝線に矢印頭を付けない

「なぜ」と「どうする」を同じ木に混ぜません。

### 8.3 因果図

用途：原因から結果までの仕組み。

構成：

- 原因
- 中間過程
- 結果
- 条件、反証、確認状況

原因から結果へ一本飛ばしせず、中間過程を書きます。

### 8.4 スイムレーン

用途：部門間の業務、情報、責任の移管。

構成：

- レーン＝担当
- 箱＝行動
- 矢印＝引き継ぎ
- ラベル＝渡すもの

矢印がレーンをまたぐ場所が、連携リスクや待ち時間の候補になります。

### 8.5 ロードマップ

用途：時間、実施順、依存関係。

構成：

- 横軸＝時間
- 行＝領域または担当
- 箱＝施策
- 接続線＝本当に依存する場合のみ

時系列は位置で表せるため、全項目を矢印でつなぐ必要はありません。

### 8.6 Before／After

用途：変化、改革、導入効果。

構成：

- 同じ比較軸
- 同じ位置
- 差分を強調
- 中央の移行矢印は一つまで

BeforeとAfterで項目や粒度を変えないでください。

### 8.7 マトリクス

用途：二軸評価、ポートフォリオ、優先順位。

構成：

- 独立した二軸
- 軸の増加方向
- 象限の意味
- 配置根拠

象限間を矢印で移動させる場合、移行可能性と条件を説明します。

### 8.8 フライホイール

用途：強化ループ、継続的な循環。

構成：

- 各要素が次の要素を強める
- 最後の要素が最初へ戻る
- 作用を動詞で説明できる

単に施策を円形に並べただけなら、プロセス図へ戻します。

## 9. 配置と余白

### 9.1 矢印より先に整列

1. 図形の粒度をそろえる
2. サイズをそろえる
3. グリッドへ置く
4. 整列・均等配置
5. 関係を確認する
6. 最後にコネクタを付ける

### 9.2 接続距離を短くする

関係が強い要素は近くに置きます。

長い矢印で遠くの要素を接続する前に、配置を変更できないか確認します。

### 9.3 線の上に文字を重ねない

ラベルは線の近くに置きますが、線を文字の中央へ通さないでください。

### 9.4 余白を流れに使う

工程間の距離は、段階差や時間差として読まれます。

意味のない不均等な間隔を避けます。

### 9.5 強調経路は一つ

全経路を同じ濃さで描き、今回説明する経路だけを強調色にするなど、視線の入口を作ります。

## 10. PowerPointでの実装

### コネクタを使う

通常の直線ではなく、図形へ接続できるコネクタを使用します。

### 整列と均等配置

目視で少しずつ動かす前に、PowerPointの整列機能を使います。

### グループ化は最後

位置と接続を確定する前にグループ化すると、修正しにくくなります。

### 接続位置をそろえる

同じ列の箱は、同じ側・同じ高さから線を出します。

### 線種を増やさない

一枚の中では、通常線、補助線、強調線程度に絞ります。

### 初期値の目安

絶対的な規則ではありません。

- 通常線：細め
- 強調線：通常線より一段太く
- 矢印頭：小さめ
- 角丸：弱め
- 影：原則なし
- グラデーション：意味がある場合のみ

図形の立体感より、関係の読みやすさを優先します。

## 11. 台本を先に作る

最終的な説明は、図形や箇条書きではなく、話し言葉として届けられます。

そのため、ストーリーボードの段階で、各ページの台本を文章として書きます。

### 必須成果物

PowerPointを制作する場合、各スライドのノート欄へ、実際に説明する文章を入れます。

PDFだけを納品する場合は、次のいずれかを付けます。

- ページ番号と対応した台本MD／DOCX
- ノート付きPDF
- 台本を収録した付録

### 台本は箇条書きではなく文章

台本の中心部分は、読み上げ可能な完全な文章にします。

メモの断片だけで終えないでください。

### 台本テンプレート

```text
このページの目的：

台本：
前のページでは〜を確認しました。ここでは〜を見ます。図の左側は〜、中央は〜、右側は〜を表しています。AからBへの矢印は〜という関係です。このため、今回の判断は〜です。ただし、〜は未確認です。次のページでは〜を具体的に確認します。

図の読み方：

強調する箇所：

前ページからの接続：

次ページへの接続：
```

「台本」の欄は、自然な話し言葉の段落として書きます。

### 台本に入れるもの

- ページの結論または問い
- 図のどこから読むか
- 各図形の役割
- 矢印が表す関係
- 数字の意味
- 条件、限界、未確認事項
- 前ページとの接続
- 次ページへの橋渡し

### スライド上の文字を読み上げない

スライドと同じ文章をそのまま読むのではなく、次を補います。

- 背景
- 解釈
- 因果
- 具体例
- 留保
- 判断

### 図と台本の対応

図形や矢印を一つ追加したら、台本で説明されるか確認します。

台本で一度も触れない図形は、削除候補です。

台本で重要な関係なのに図へ現れない場合、図を修正します。

### 視覚情報を言葉にする

色、位置、矢印、写真だけで伝えず、重要な視覚情報を台本でも言葉にします。

例：

> 赤い経路は、現時点で最も待ち時間が長い引き継ぎです。

> 左から右へ、問い合わせの受付、分類、回答の順に進みます。

これは説明の明確さだけでなく、視覚情報へアクセスしにくい読み手への配慮にもなります。

---

## 12. 制作工程への統合

### STEP 5：ストーリーボード

各ページについて、次を作ります。

- ページの役割
- 結論または問い
- 使用する原材料
- 図の一文
- 読み上げ台本の初稿
- 前後ページとの接続

この段階では、完成図形を作りません。

### STEP 7：図の文法

- 図形の役割
- 矢印の動詞
- 囲み・レーンの意味
- 色の意味
- 主方向
- 線種

を先に定義します。

### STEP 8：代表ページ

代表ページでは、見た目だけでなく、実際に台本を読み上げて確認します。

- 視線と説明順が一致するか
- 矢印を自然に説明できるか
- 図を見ずに台本だけ読んでも論理が通るか
- 台本を聞きながら図を追えるか
- 文字と話が重複しすぎていないか

### STEP 10：最終レビュー

全ページのノート欄を確認します。

台本のないスライドを完成扱いにしません。

## 13. レビュー項目

### 図の意味

- [ ] 図形が何を表すか説明できる
- [ ] 各矢印を動詞で言える
- [ ] 同じ形は同じ役割に使われている
- [ ] 同じ線種は同じ関係に使われている
- [ ] 比較や並列へ不要な矢印を使っていない

### 流れ

- [ ] 主方向が一つ
- [ ] 始点と終点が明確
- [ ] 分岐条件が書かれている
- [ ] 線が交差していない
- [ ] 曲がりが多すぎない
- [ ] 循環に本当の戻りがある

### 配置

- [ ] 図形のサイズと粒度がそろっている
- [ ] 整列と均等配置ができている
- [ ] 強い関係ほど近くにある
- [ ] 線が文字を横切っていない
- [ ] 強調経路が一つに絞られている

### 台本

- [ ] 各スライドに文章形式の台本がある
- [ ] 台本がスライド本文の読み上げになっていない
- [ ] 図の読み順を説明している
- [ ] 主要な矢印の関係を言葉にしている
- [ ] 条件、限界、未確認事項がある
- [ ] 前後ページへの接続がある

## 14. 参照した考え方

- ISO／ANSI系フローチャート：形と線へ役割を割り当てる
- UML Activity Diagram：行動、判断、分岐、合流を限定した記号で表現する
- Swimlane：担当と引き継ぎを分ける
- Issue Tree：問いを一方向へ分解する
- Functional Flow Block Diagram：左から右の機能的な流れ
- IDEF0／SADT：入力、出力、制御、手段によって矢印の意味を分ける
- Gestalt principles：近接、類似、連続、共通領域による関係づけ
- Multimedia learning：画面上の情報と口頭説明の重複、視線移動、注意誘導を管理する
- Presentation accessibility：視覚要素を言葉でも説明する
