# Wordマニュアル制作ガイド

> 状態：stable  
> designmd版：2026.07.21-beta  
> 最終更新：2026-07-21

この文書は、業務手順、引継ぎ、定期運用、システム操作などをWordでマニュアル化するときの判断基準を定める。

> マニュアルは、一般的に正しそうな説明を網羅する文書ではない。  
> この業務で実際に迷う場所、事故が起きる場所、例外時の判断、責任と参照先が分かる作業道具である。

誰か一人の後任だけを想定せず、業務を担当する人が同じ判断へ到達できる文書を目指す。

---

## 1. 最優先する情報

マニュアルでは、次の順番で情報を扱う。

1. 実際の作業順序
2. 過去に起きた、または起きかけた事故
3. 重要度や難易度が高い工程
4. 担当部署、責任主体、現在の作業窓口
5. 例外時の過去事例と採用理由
6. 作業中に使うリンク、連絡先、ファイル
7. 未確認事項、改善案

一般的なベストプラクティスは、実際の運用として確認できた場合だけ標準手順へ入れる。

### 書いてよいもの

- 現在行っている手順
- 実際に使う画面、ファイル、設定値
- 過去に起きたミス
- 現場で重要と認識されている注意点
- 実在する担当部署、担当領域、作業者
- 例外時に採用した過去事例
- 判断できなかった点

### 勝手に標準手順へ入れないもの

- ユーザーが語っていないチェック運用
- 実施確認できていない保存、承認、履歴管理
- 一般論として望ましいだけのダブルチェック
- 使われている根拠がない月次チェックシート
- 実務上不要な網羅的分類
- AIが補った担当者、期日、確認項目

追加を提案したい場合は、本文へ混ぜず、**改善案**または**確認したい点**として分ける。

## 2. 分類名より、事故の姿を書く

事故ポイントを「対象者」「URL」「システム」のような分類名だけで終わらせない。

### 弱い例

```text
・対象者を確認する
・URLを確認する
・配信設定を確認する
```

### 強い例

```text
・新しく加わった対象者が漏れていないか
・対象外になった人が前月の一覧に残っていないか
・プログラムだけ更新し、メール本文が前月のままになっていないか
・案内URLが前回分のまま残っていないか
```

分類は検索や整理に使える。ただし、現場で事故を防ぐのは、**何がどう間違うかを想像できる文**である。

### 事故情報の書式

```text
【起きること】

【なぜ起きるか】

【どこで確認するか】

【間違えた場合の影響】
```

すべての事故へ四項目を機械的に付けない。影響が大きい、頻度が高い、見落としやすい事故だけ詳しく書く。

---

## 3. 説明量を均等にしない

AIが作るマニュアルは、各STEPを同じ分量、同じ温度で説明しやすい。実務では、危険度、難易度、事故頻度に差がある。

### 説明を濃くする場所

- 対象や期間を決める設定
- 外部公開、配信、削除、上書きに関わる操作
- 前月や過去データを流用する工程
- 複数画面や複数ファイルをまたぐ工程
- 担当部署が分かれ、受け渡しが発生する工程
- 過去に事故が起きた工程

### 説明を薄くできる場所

- 画面を見れば結果が分かる単純操作
- 毎回同じで、判断を伴わない操作
- 他のSTEPと同じ内容の再説明

「重要」とだけ書かず、理由を続ける。

```text
重要
この設定は配信対象と実行期間を決めるため、前月値の残存に注意する。
```

---

## 4. 同じ情報を三重化しない

同じ確認事項を、本文、表、チェックリストで繰り返さない。

### 情報の基本配置

- **手順**：本文
- **日付、設定値、比較条件**：表
- **重大事故**：該当STEPの注意欄
- **例外**：該当STEPの直後
- **リンク、連絡先**：使う場所の近く
- **最終確認**：実際に運用されている項目だけ

冒頭に重大事故の一覧を置く場合、本文で全文を再掲しない。該当STEPへの参照か、一行の注意にとどめる。

チェックリストを作るのは、実際に作業時にチェックを付ける運用がある場合、または重大な確認を作業終了時にまとめて行う場合だけにする。

---

## 5. 担当部署、役割、個人名を使い分ける

担当者名を機械的に部署名へ置き換えない。逆に、すべてを個人名で固定しない。

### 基本

- **責任主体**：部署、チーム、担当領域
- **現在の作業窓口**：個人名
- **承認者**：役職または担当範囲

```text
担当：チーム広報
現在の作業窓口：○○さん
```

個人名が変わっても業務責任が残る場合、部署や役割を先に書く。特定の人しか作業できない現状が重要な場合は、個人名も消さない。

公開リポジトリや外部共有版には、個人名、メールアドレス、内部URL、未公開情報をそのまま載せない。社内版と公開版を分ける。

---

## 6. 例外は、過去の実例で残す

例外処理を「関係者と相談する」「代替案を検討する」で終わらせない。

### 残す内容

- どの条件で通常手順を使えなかったか
- 過去に何を代替案として採用したか
- なぜその案を選んだか
- どこまで再利用できるか
- 今回も同じ判断でよいか未確認か

```text
例外
対象者がいない場合は、過去の代替企画と採用理由を確認する。
前回例：○○
採用理由：○○
今回は未確認：○○
```

過去事例は、単なる思い出ではない。次の担当者が判断を再現するための**組織の記憶**として扱う。

---

## 7. リンク、連絡先、ファイルは使う場所へ置く

巻末の参照リンク一覧だけに依存しない。

### ルール

- 管理表は、対象者を確認するSTEPへ置く
- 問い合わせ先は、問い合わせが発生するSTEPへ置く
- ページURLは、差し替えや確認を行うSTEPへ置く
- 同じリンクを何度も長いURLで再掲しない
- 表示名から用途が分かるようにする

```text
管理ファイル：誕生日バナーリスト
用途：対象月、コメント、公開URLの確認
```

印刷利用がある場合は、クリック可能なリンクに加えて、短縮しすぎない参照名や保存場所も示す。

---

## 8. 現場のコメントを消しすぎない

「ここが肝」「本来の担当部署」「前回ここで間違えた」のようなラフなコメントには、説明量の強弱や責任の実態が含まれている。

そのままの口調を必ず残す必要はない。ただし、情報を一般論へ丸めない。

### 変換例

```text
修正前：
⇒ここがオートメーションの肝の部分

マニュアル表現：
重要
このSTEPは対象と実行期間を決める中心設定のため、前月値が残っていないか確認する。
```

体裁は整える。現場の重みは消さない。

## 9. Wordの基本デザイン

Wordマニュアルは、白地、黒または濃いグレーの文字を基本にする。色でにぎやかにしない。

### 基本設定

- 用紙：A4縦を基本とする
- 余白：既存社内テンプレートを優先。指定がなければ上下左右18〜22mm程度
- 本文書体：BIZ UDPゴシック。利用できない場合は游ゴシック
- 本文サイズ：10.5〜11pt
- 行間：1.2〜1.35程度
- 見出し：Wordの組み込みスタイルを使う
- ページ番号：複数ページではフッターへ表示する
- 表紙：文書名、対象業務、版、更新日だけを基本とする

数値は固定テンプレートではない。既存の社内書式や読み手の年齢、印刷環境がある場合はそちらを優先する。

### 色

- ベース：白、黒、濃いグレー
- 赤：重大事故、禁止、必須確認、高リスク設定
- その他の色：意味があり、案件側の既存ルールがある場合だけ

赤文字を色だけの合図にしない。必ず「重要」「注意」「禁止」などのラベルや文言を併記する。

```text
重要
メール本文が前月のまま残っていないか確認する。
```

### 強調

強調を乱発しない。

- 一つの節に、主となる強調は原則一つ
- 太字、赤字、網掛け、囲みを同じ箇所へ重ねない
- すべてのSTEPへ「重要」を付けない
- 長い段落全体を赤字にしない
- 黄色マーカーを装飾として使わない
- 見出しだけで階層が分かる場所へ、追加の囲みを置かない

安全、法令、重大な公開事故を扱う場合は、節一つという上限より必要な警告を優先する。ただし、警告の種類を整理する。

### 強調の優先順位

1. 見出しと位置
2. 短い太字
3. 注意ラベル
4. 赤文字
5. 囲み、網掛け

色より先に、構造と文章で重要度を伝える。

## 10. 見出しとページ構造

Wordの見出しは、見た目だけ大きくせず、組み込みの「見出し1」「見出し2」「見出し3」を使う。

### 理由

- ナビゲーションウィンドウで移動できる
- 自動目次を作れる
- 読み上げ時に構造が伝わる
- 全体の書式を一括変更できる

### ルール

- 一つの見出しに複数の話題を詰めない
- 見出しだけをページ末尾へ残さない
- 見出し直後に一行以上の説明または手順を置く
- STEP番号が飛んでいないか確認する
- 項番と見出し階層を混同しない
- 長い文書だけ目次を使う

目次はページ数を立派に見せるために作らない。ナビゲーションが必要な長さや利用頻度がある場合だけ使う。

## 11. 表、スクリーンショット、図

### 表

表は、同じ軸で比較する情報へ使う。

向くもの：

- 日付と設定値
- 役割と担当範囲
- 状態と対応
- 複数環境の違い

向かないもの：

- 手順本文の再掲
- 長い説明文
- 一つしか値がない情報
- セル内に複数の判断を詰めたもの

複数ページへ続く表では、見出し行を繰り返す。セル内の文字を小さくして無理に一ページへ収めない。

### スクリーンショット

- 画面全体ではなく、操作対象が分かる範囲へ切り取る
- 一枚の画像へ多数の番号を付けない
- 番号と本文の対応を近くに置く
- 文字が読めない縮小画像を使わない
- 操作後の状態が重要なら、操作前後を分ける
- UI更新で古くなる可能性が高い画像には更新日を持たせる
- 代替説明またはキャプションを付ける

### 図

図は、画面関係、分岐、受け渡し、例外処理など、本文だけでは関係を追いにくい場合に使う。本文を箱と矢印へ入れ直すだけなら使わない。

## 12. 推奨構成

すべてのマニュアルを同じ章立てにしない。業務に必要な項目だけ使う。

```text
1. 目的と対象範囲
2. 作業前に確認すること
3. 重大事故または最重要ポイント
4. 実際の作業順に沿ったSTEP
5. 各STEPの例外、担当、リンク
6. 実際に使われている最終確認
7. 更新履歴
```

### 短いマニュアル

- 目的
- 手順
- 重要な注意
- 参照先

### 月次、定期運用

- いつ始めるか
- 最新一覧をどこで確認するか
- 前月流用で起きる事故
- 月ごとに変わる設定
- 例外時の過去例
- 完了条件

### システム操作

- 対象画面への入口
- 変更する値
- 変更しない値
- 保存、公開、実行のタイミング
- 実行前の重大確認
- 実行後に確認する状態

## 13. 制作工程

### PASS 0：原材料を分ける

```text
【確認済みの現行手順】

【実際の事故・ヒヤリハット】

【担当部署・担当領域】

【現在の作業窓口】

【例外と過去事例】

【リンク・ファイル・連絡先】

【未確認事項】

【改善案】
```

未確認事項と改善案を、現行手順へ混ぜない。

### PASS 1：実作業の順へ並べる

組織図、システム分類、ファイル分類ではなく、担当者が行う順番を優先する。

### PASS 2：危険度で濃淡を付ける

すべてを均等に説明せず、事故が起きる場所へ具体例と理由を置く。

### PASS 3：重複を削る

本文、表、チェックリスト、巻末で同じ内容を繰り返していないか確認する。

### PASS 4：Wordの構造を整える

見出しスタイル、改ページ、表、画像、リンク、強調を整える。内容を一般論へ戻さない。

### PASS 5：実寸確認

- Wordで開いた状態
- PDF化した状態
- 白黒印刷
- 100％表示
- ナビゲーションウィンドウ
- リンクのクリック
- ページ番号、改ページ、表の分断
- アクセシビリティチェック

を必要な範囲で確認する。

## 14. 最終レビュー

### 内容

- [ ] 実際の作業順になっている
- [ ] 一般論より、実際の事故が優先されている
- [ ] 未確認の運用を標準手順へ昇格させていない
- [ ] 重要なSTEPだけ説明が濃くなっている
- [ ] 部署、役割、個人名の使い分けが実態と合っている
- [ ] 例外へ過去事例または判断材料がある
- [ ] リンク、連絡先、ファイルが使う場所の近くにある
- [ ] 同じ情報を本文、表、チェックリストで繰り返していない
- [ ] 現場のコメントを一般論へ丸めていない
- [ ] 見出し番号、用語、部署名、システム名がそろっている

### Word

- [ ] 色を多用していない
- [ ] 赤は重大な注意だけに使っている
- [ ] 一節の強調が多すぎない
- [ ] 色だけで意味を伝えていない
- [ ] Wordの見出しスタイルを使っている
- [ ] ページ末尾に見出しだけが残っていない
- [ ] 表や画像の文字が実寸で読める
- [ ] リンクがクリックでき、用途が分かる
- [ ] 画像へキャプションまたは代替説明がある
- [ ] アクセシビリティチェックを必要な範囲で行った

## 15. AIへ渡す指示

```text
以下の原材料からWordマニュアルを作成してください。

必須要件：
- 一般的なベストプラクティスを、確認なしで現行手順へ追加しない
- 確認済み手順、実際の事故、例外、担当、リンク、未確認事項を分ける
- 実際の作業順に並べる
- 同じ情報を本文、表、チェックリストで繰り返さない
- 危険度、難易度、事故頻度に応じて説明量へ差を付ける
- 部署・役割を責任主体、個人名を現在の窓口として使い分ける
- リンクや連絡先は使うSTEPの近くへ置く
- 色は白、黒、濃いグレーを基本とし、赤は重大な注意だけに使う
- 強調は一節につき主となるものを原則一つにする
- Wordの見出しスタイルを使い、ナビゲーション可能にする
- 不明点は推測で補わず、「未確認事項」または「改善案」へ分ける

原材料：
（ここに貼る）
```
