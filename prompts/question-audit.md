# 問いの監査プロンプト

制作へ入る前に、与えられたブリーフの前提をAIと人間で監査する。

```text
以下のガイドを参照してください。

共通原則：
https://raw.githubusercontent.com/silovar-uk/designmd/main/DESIGN.md

最初の一問：
https://raw.githubusercontent.com/silovar-uk/designmd/main/docs/question-blocks.md

問いの監査：
https://raw.githubusercontent.com/silovar-uk/designmd/main/docs/question-audit.md

# 0. 最初の一問

最初の応答では、監査結果や代替問いを出す前に、監査の方向を最も変える一問だけを聞いてください。

- ブリーフと会話から、既に分かっていることは聞き直さない
- 読み手、使用場面、表面的な課題の先で変えたい状態などから、影響が最大の未知を一つ選ぶ
- 題材に合う2〜3案、推奨案、短い理由を添える
- 疑問文と質問符を一つだけにする
- 回答前に監査へ進まない
- 回答後に追加質問をしない

質問例：

最初に一つだけ確認します。
今回、問い直しによって最も変えたいのは A｜読み手の理解、B｜組織の判断、C｜実際の行動 のどれですか？ 制作へつなげるため、推奨はBとCです。

回答後は、残る不足を置いた前提、未確認事項、追加調査候補として管理し、推奨案で監査を完了してください。

# 目的

最初の一問の回答を踏まえ、与えられた問いへすぐ答えず、前提、根拠、本当に変えたい状態、外部文脈、代替問いを確認してください。

AIは問いを問い直して構いません。ただし、奇抜な問いや反転案を出すこと自体を目的にしないでください。採用する問いの最終判断は人間が行います。

# 問いの監査

1. 前提
   ブリーフが事実として扱っている内容を抽出してください。

2. 根拠
   各前提を、確認済み事実／依頼者の仮説／慣習的な表現／未確認に分類してください。

3. 欲望
   表面的な課題が解決した先で、依頼者は何を変えたいと考えられるか、複数仮説を示してください。断定しないでください。

4. 文脈
   生活者、競合、組織、社会、時代の観点から、不足している材料を示してください。確認できない事実を創作しないでください。

5. 代替問い
   次の三案を作ってください。
   - ブリーフ忠実案
   - 欲望へ持ち上げた再定義案
   - 前提を採用しない反転・別軸案

# 比較

三案を、次の観点で比較してください。

- 根拠
- 得られるもの
- 失うもの
- 実装可能性
- 誤解・倫理・組織上のリスク

# 禁止

- 「常識を疑う」「逆転の発想」などの定型だけで問いを変えない
- 社会課題へ接続することを自動的に上位概念としない
- 根拠なく「本当の課題は別にある」と断定しない
- 依頼者の目的を理解せず、ブリーフを否定しない
- 緊急、安全、法令、決定済み事項を面白さのために広げない
- 不足情報が見つかったたびに追加質問をしない

# 出力

1. 最初の一問と回答
2. 回答によって変えた監査条件
3. 依頼された問い
4. 前提と根拠の状態
5. 本当に変えたい状態の仮説
6. 不足している外部文脈
7. 三つの代替問い
8. 比較表
9. 推奨する問いと理由
10. 採用時に必要な追加調査
11. 置いた前提と未確認事項
12. 人間が最終判断すべき点

ブリーフ：
（ここに貼る）
```
