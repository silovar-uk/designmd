# 逆質問テキストブロック用プロンプト

制作前に不足情報を確認するとき、通常の箇条書きではなく、次の形式で返してください。

空欄だけを並べて、ユーザーへゼロから考えさせないでください。

会話から読み取った前提と仮説を先に示し、方向性に関する質問では2〜4案、推奨案、推奨理由を提示してください。ユーザーは違うところだけ修正できる形にします。

```text
【すでに確定している前提】
- テーマ：
- 成果物：
- 既に分かっている要望：
- 過去のフィードバック：

【こちらの仮説】
- 

空欄は推奨案で進めます。違うところだけ書き換えてください。

【必須】
1. 主な読み手：
   こちらの仮説：
   他の候補：
   推奨：
   理由：
   回答：推奨で進める／修正：

2. 使用場面：
   A｜
   B｜
   C｜
   推奨：
   理由：
   回答：推奨で進める／修正：

3. 読後に求める理解・判断・行動：
   こちらの仮説：
   推奨：
   理由：
   回答：推奨で進める／修正：

【推奨】
4. 今回いちばん出したい温度：
   A｜創業者の原体験
   B｜プロダクトローンチ
   C｜実験発表
   D｜チームのキックオフ
   推奨：
   理由：
   回答：推奨で進める／修正：

5. ユーモアの出し方：
   A｜欄外の小さなツッコミ
   B｜自分たちの失敗を見せる自虐
   C｜構成そのものを一度だけ崩す
   推奨：
   理由：
   回答：推奨で進める／修正：

6. 実例や失敗の扱い：
   A｜実物を大きく見せる
   B｜再現図にする
   C｜文章だけで触れる
   D｜出さない
   推奨：
   理由：
   回答：推奨で進める／修正：

【任意】
7. 参考にしたい媒体や雰囲気：
   こちらの仮説：
   回答：その通り／追加：

8. 絶対に避けたい表現：
   こちらの仮説：
   回答：その通り／追加：
```

ルール：

- 既に分かっていることは、質問せず「すでに確定している前提」に入れる。
- 仮説と確定事項を混同しない。
- 方向性の質問では、今回の題材に合わせた2〜4案を出す。
- 選択肢だけを並べず、推奨案と推奨理由を示す。
- 回答欄は `推奨で進める／修正：` を基本にする。
- `その他` や案の組み合わせを許可する。
- 日時、予算、人数、権利条件など、誤ると危険な事実は無理に推測しない。
- 必須、推奨、任意の三分類にする。
- 質問は最大8問。
- 回答前に完成版の制作へ進まない。
- ユーザーが「質問不要」「一括で」と明示した場合のみ、質問を省略する。
