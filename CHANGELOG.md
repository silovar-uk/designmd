# CHANGELOG

このファイルは、designmdガイド群に対する大規模な変更の記録であると同時に、今後この種の変更を行う際の運用ルールを定めるドキュメントである。個々のコミット履歴の代わりではなく、「なぜこの構成になっているか」「次に変更するときは何を確認すべきか」を残す場所として使う。

対象読者は、このリポジトリのdocs/やprompts/を編集するAI・人間の両方である。ガイドの利用者（`DESIGN.md`のURLだけを外部AIへ渡して使う立場）は、原則としてこのファイルを読む必要はない。

## 2026年：GATE制からSTEP制への再設計

### 背景

既存のガイド群は、「AIくささの回避」（禁止語、均質な構造、抽象語過多などを検出し、避ける）には強かった。しかし、それを守るだけでは出力が「AIくささはないが、誰も本気で選んでいないように見える、きれいなAI資料」の域を出ないという課題があった。悪さを避ける規則はあっても、良さをどう作るかという正典が薄く、物語としての必然性、見せ場の設計、資料タイプごとの方針分岐が体系化されていなかった。

また、制作工程の呼称が `docs/workflow.md` の GATE 0–8 と `docs/staged-production.md` の記述とで並立し、資料間で参照する段階番号が揺れていた。加えて、段階制作の説明、参照戦略の要点、あたたかみの原則などが3〜4ファイルにまたがって再掲され、一方を直しても他方が古いまま残るという重複による劣化が起きていた。

### 変更点の要約

- **workflow.md統合**：`docs/workflow.md` を廃止し、GATE 0–8 の内容を `docs/staged-production.md` の統一STEP 1–10工程へ再編した。以後、制作工程の正典は `staged-production.md` 一本のみとする。
- **新規4ファイル**：
  - `docs/story-design.md`（物語設計：冒頭の型、転換、見せ場の間隔、密度リズム譜、タイトルの選び方）
  - `docs/document-type-policies.md`（資料タイプ別方針：発表資料・社内報告・学習資料・その他で異なる既定値を判定する）
  - `docs/ai-failure-patterns.md`（AI失敗パターン集：症状・原因・代替行動のカタログ、および案件間バリエーション規則）
  - `docs/quality-rubric.md`（品質ルーブリック：成果物10軸・ガイド自体5軸の到達水準評価）
- **visuals.md拡張**：60行から301行へ拡張し、タイポグラフィ・レイアウト・写真・色のビジュアル技法の正典として再定義した。
- **core-principles.md書換**：理念、4つの資質（熱量・抑制・明晰さ・緩急）の統合、AIくささの9要因診断を一本化した正典へ書き換えた。
- **重複削減**：`docs/slides.md`、`docs/writing.md` を減量し、他正典への参照へ置き換えた。`docs/playful-editoriality.md`、`docs/question-blocks.md`、`docs/learning-materials.md` の二次テンプレート化（本来は原則集であるべき箇所が、そのまま流用できる定型文になっていた状態）を解消した。
- **改番**：4つのレビューチェックリストと `prompts/` 配下のプロンプト群をSTEP 1–10表記へ統一し、`docs/quality-rubric.md` への接続を追加した。
- **DESIGN.md再生成**：465行で、docs正典群の内容を自己完結的にダイジェストした版へ全面更新した。GATE表記は完全に除去した。

### Before / After

**Before**：悪さの回避には強いが、良さの作り方が欠落している。GATE番号がファイル間で不統一。段階制作・参照戦略・あたたかみなどの説明が同じ内容のまま3〜4ファイルに再掲され、更新漏れが起きやすい。

**After**：物語設計・ビジュアル技法・資料タイプ別方針という「良さの作り方」が正典として追加された。制作工程はSTEP 1–10に統一された。正典所有制により、各概念は1ファイルにのみ存在し、他は要約と参照リンクにとどまる。

## 今後の更新ルール

このリポジトリを継続的に良い状態へ保つため、以後の変更では次を守る。

### 1. 追加前に正典を確認する

新しい原則やルールを追加したくなったら、まず既存の正典ファイルにその概念がすでにないか確認する。特に次のファイルは、それぞれ固有の概念領域を持つ正典であり、似た内容を書きたくなったらまずここを見る。

- 理念・4資質・9要因診断 → `docs/core-principles.md`
- 制作工程（STEP 1–10） → `docs/staged-production.md`
- 逆質問の型 → `docs/question-blocks.md`
- 4層分離・機能保存表 → `docs/audience-first-redesign.md`
- 物語設計 → `docs/story-design.md`
- ビジュアル技法 → `docs/visuals.md`
- 資料タイプ別方針 → `docs/document-type-policies.md`
- 失敗パターン・案件間バリエーション規則 → `docs/ai-failure-patterns.md`
- 品質ルーブリック → `docs/quality-rubric.md`
- 参照戦略 → `docs/reference-strategy.md`

該当する正典がすでにある場合は、新規ファイルを作らず、その正典を更新する。存在しない新しい概念領域であることを確認できた場合のみ、新規追加を検討する。

### 2. 概念の正典は1つに保つ

ある概念について書く必要が別ファイルで生じた場合も、そこに全文を書かない。1〜3行の要約と、正典ファイルへのリンクにとどめる。正典側の記述を書き換えたときに、要約側も意味がずれていないか合わせて確認する。

### 3. docs/を編集したらDESIGN.mdも同期する

`docs/` 配下の正典ファイルを編集した場合、`DESIGN.md` の対応する章を同じ内容へ更新する。DESIGN.mdは外部AIへ渡す唯一の自己完結ダイジェストであり、ここが古いままだと、リポジトリ内では直っているのに外部からは古い内容が参照され続ける。

### 4. STEP番号を変更する場合は参照元をすべて洗い出す

`docs/staged-production.md` のSTEP番号や順序を変更する場合、次の参照元を必ず洗い出して更新する。

- 4つのレビューチェックリストが持つ「戻る段階」の記述
- `prompts/` 配下のプロンプトが引用するSTEP番号
- `DESIGN.md` の対応箇所
- `docs/story-design.md`、`docs/document-type-policies.md` など、特定STEPから参照されると明記している他の正典ファイル

STEP番号は複数ファイルから逆参照される背骨であるため、1箇所だけ直して他を放置すると、資料間でSTEP番号が食い違う状態（今回の再設計前と同じ問題）に戻る。

例：STEP 7（ビジュアル方針）とSTEP 8（レイアウトと文字組み）を統合してSTEP番号を1つ減らす場合、それ以降のSTEP 9・10を指している全ファイルの番号もずれる。番号を振り直す前に、`grep`等で「STEP 8」「STEP 9」「STEP 10」を含むファイルを洗い出してから着手する。

### 5. 案件間バリエーション規則を自己適用する

`docs/ai-failure-patterns.md` が定める案件間バリエーション規則（同じ症状でも媒体や題材によって現れ方が変わる、という前提）は、ガイド自体にも適用される。ガイドの例文・数値パターンをそのまま繰り返し使うと、その例自体が新しい「型」として模倣される対象になり、ガイドが警告している均質化を、ガイド自身が引き起こすことになる。例を追加・更新する際は、題材、数値、業種などの多様性を意識する。

例：失敗パターン集の具体例が毎回「3項目の箇条書きを3枚のカードにする」という同じシナリオばかりだと、読み手はそのシナリオだけを覚え、他のシナリオ（5項目を5レーンの図にする、比較表を無理に矢印でつなぐ、など）で同種の判断ミスを避けられなくなる。業種・資料タイプ・項目数を変えた例を混在させる。

### 6. raw URLで外部参照されているファイルの名前・パスを変更しない

次のファイルは、README.mdやAGENTS.md内のプロンプト例文で `https://raw.githubusercontent.com/...` として外部のAIツールへ直接渡される想定になっている。ファイル名やパスを変更すると、既にこのURLを使っているユーザーの参照が壊れる。

- `DESIGN.md`
- `docs/question-blocks.md`
- `docs/slides.md`
- `docs/plain-slides.md`
- `docs/diagram-shapes.md`
- `docs/visuals.md`
- `docs/staged-production.md`
- `docs/reference-strategy.md`
- `docs/playful-editoriality.md`
- `docs/story-design.md`
- `docs/document-type-policies.md`
- `docs/ai-failure-patterns.md`
- `docs/quality-rubric.md`
- `docs/audience-first-redesign.md`
- `docs/learning-materials.md`

改名がどうしても必要な場合は、（a）旧ファイル名をリダイレクト用に残す（内容を新ファイルへの案内だけにする）か、（b）README.md・AGENTS.md・`.github/copilot-instructions.md`・`site/index.html`・`prompts/` 配下・`DESIGN.md` を含む参照元をすべて更新してから削除する、のいずれかを行う。

### 7. 新しいファイルはREADME.mdの層構造に位置づける

新しいdocsファイルを追加する場合、追加する前に、README.mdの5層構造（第1層 思想／第2層 プロセス／第3層 技法／第4層 媒体・タイプ別／第5層 検証）のどこに属するかを決める。層が決まらないファイルは、既存の正典との重複を疑う。層が決まったら、README.mdの該当層の表、および正典所有表（該当する場合）へ追加する。

### 8. 「必ず」「常に」の使用を制限する

例外条件を伴わない「必ず」「常に」は、新規追加を避ける。本ガイドは「内容上の理由があればルールを破ってよい」という前提を持つため、無条件の断定はこの前提と矛盾する。既存の記述に見つけた場合も、書き換えの機会があれば例外条件付きの表現へ直す。

## このファイルの更新方法

次回以降、design.md群に対して同程度の規模（複数ファイルの新設・再編・改番を伴う）の変更を行った場合は、「2026年：GATE制からSTEP制への再設計」と同じ形式で新しい節をこのファイルの先頭（このセクションの前）に追加する。

1. 見出しは `## <年>年：<変更を一言で表す名前>` とする。
2. 背景・変更点の要約・変更ファイル一覧・Before/Afterの4項目を最低限含める。
3. 「今後の更新ルール」節は、既存ルールと矛盾しない限り上書きせず、必要な場合のみ追記・改訂する。ルール自体の変更履歴は、ルール本文の下に「（2027年改訂：〜）」のように注記すれば足り、独立した節を新設する必要はない。

軽微な誤字修正や1ファイル内で完結する小さな変更は、このファイルへ記録しなくてよい。CHANGELOG.mdは複数ファイルにまたがる構造的な変更を対象とする。
