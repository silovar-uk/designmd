# Repository instructions for AI agents

このリポジトリで文章、スライド、画像、プロンプト、サイトを追加・修正するAIは、最初に [`DESIGN.md`](DESIGN.md) を読むこと。

現在版は [`VERSION`](VERSION)、正典の状態と更新方法は [`docs/governance.md`](docs/governance.md)、大きな変更は [`CHANGELOG.md`](CHANGELOG.md) を確認する。

## 作業時の必須事項

1. 読み手、使用場面、読後の変化、判断事項を明示する。
2. 事実、解釈、仮説、提案、演出を混同しない。
3. 原材料の具体性を残す。
4. 完成版を一度に作らず、統一STEP 1–10を通す。
5. 対話可能な場合は、逆質問、構成案、代表ページで一度止まる。
6. 逆質問は `text` コードブロックで提示する。
7. 空欄だけを並べず、確定事項、仮説、具体案2〜4案、推奨案、推奨理由を先に示す。
8. 回答欄は `推奨で進める／修正：` を基本にする。
9. 日時、人数、予算、権利条件など、誤ると危険な事実は推測しない。
10. 構成案を最低3案比較し、採用理由と不採用理由を示す。
11. designmdの共通基盤と、案件固有のスタイルガイドを分ける。
12. 案件固有の書体、色、余白、語り口を、共通テンプレートとして固定しない。
13. 案件別スタイルガイドはSTEP 1で仮置きし、STEP 7で確定、STEP 8で検証する。
14. アクセシビリティをSTEP 1、構成、視覚、実装、最終検証の全工程で確認する。
15. 色、位置、アニメーションだけで意味を伝えない。
16. 図、画像、表、動画の要点へ代替説明を用意する。
17. designmdやAIが、規格適合、正確性、権利、公開可否を保証するように書かない。
18. 参照先を、骨格、逸脱、時間、固有素材へ分ける。
19. 参照先から借りる操作と、借りない外見を示す。
20. 同一テンプレートを内容に関係なく反復しない。
21. グリッドを同じ箱の反復ではなく、情報間の関係として使う。
22. 静、標準、密、動の代表ページを全体制作前に確認する。
23. 独自性やダイナミックさは、秩序の上に置く一つの逸脱ルールで作る。
24. 読者向け本編、発表者用台本、付録・根拠、制作ログを分ける。
25. 制作ブリーフ、ページ役割、採用・不採用理由を、必要なく本編へ表示しない。
26. 既存成果物の再設計では、旧版の機能保存表を先に作る。
27. 新版は旧版より、理解、判断、行動、再現のいずれかを改善する。
28. 抽象概念の近くに、数字、発言、操作、実物、Before／Afterなどの証拠を置く。
29. 図は本文の再掲ではなく、前後、因果、差分、分岐、位置、変化を新たに示す。
30. 箇条書きで十分な並列情報を、カード、アイコン、矢印、図形へ無理に変換しない。
31. 図形化で新しい関係が増えない場合、文章、箇条書き、表を優先する。
32. 図形は名詞、矢印は動詞、囲み・レーンは文脈として定義する。
33. 矢印を置く前に、接続を「AがBへ何をする」と文章で説明する。
34. 並列、比較、グルーピングへ不要な矢印を使わない。
35. PowerPointでは、各スライドのノート欄へ文章形式の台本を入れる。
36. ノートの説明文は常体、だ・である調で統一する。
37. 引用以外に「です」「ます」「ございます」を残さない。
38. 常体にするため報告書調へ硬化させず、自然な口頭説明を保つ。
39. 台本には、図の読み順、背景、因果、条件、限界、前後ページへの接続を含める。
40. 学習資料では、定義、識別、理由、手順、変化、失敗条件、転用を確認する。
41. スマートフォン、PC 50％、A4、投影など、実際の閲覧条件で確認する。
42. 画像・図の情報上の役割を説明する。
43. 採用・棄却理由を制作ログへ残す。
44. `docs/quality-rubric.md` で採点し、3点以下があれば該当STEPへ戻る。
45. 完成後、`docs/review-checklist.md` に沿って点検する。

## Experimentalの扱い

`docs/format-selection.md` と `docs/type-spacing-decisions.md` はexperimentalである。

- 通常ルールと同じ強さで強制しない
- 形式や数値を先に決め、内容を当てはめない
- 3案件で試行結果を記録する
- 形式先行、同型化、制作時間増、理解改善なしの場合は撤回または縮小する
- stable化するときはVERSION、CHANGELOG、DESIGN.md、READMEを更新する

## 対話できない場合

一度の出力を求められた場合も、内部では全STEPを順番に通す。

提出時に次を示す。

- 置いた前提
- 仮の案件別スタイルガイド
- アクセシビリティ前提
- 比較した構成案と不採用理由
- 採用理由
- 参照した資料、借りた操作、借りなかった外見
- 本編、台本、付録、制作ログの分離
- 旧版から保存した機能と失われた機能
- グリッド、文字階層、密度、逸脱ルール
- 箇条書きのまま残した箇所と理由
- 代表ページで確認したこと
- 実寸、読み上げ、操作テスト
- 各ページの台本
- 未確認事項
- 最終判断者

## 保守規約

1. 各概念には一つの正典を置く。他ファイルでは要約とリンクにとどめる。
2. 新しい原則を追加する前に、既存正典との重複を確認する。
3. 新しい規範には、目的、向く場面、注意が必要な場面、例外、検証方法を書く。
4. `docs/` の正典を編集したら、`DESIGN.md` の同期要否を確認する。
5. STEP番号を変更する場合は、レビュー、プロンプト、DESIGN.mdの参照をすべて更新する。
6. 正典の追加、削除、意味変更は `CHANGELOG.md` に記録する。
7. `experimental` には評価期限、試行範囲、撤回条件を書く。
8. 版を変更する場合は `VERSION`、README、サイト表示を同期する。
9. デジタル庁デザインシステムの参考箇所を変更する場合は、`docs/references/digital-agency-design-system.md` の出典・加工表示を更新する。
10. 案件間バリエーション規則を、ガイドの例文とサイト表示にも自己適用する。
