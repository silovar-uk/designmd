# Repository instructions for AI agents

このリポジトリで文章、スライド、画像、プロンプトを追加・修正するAIは、最初に [`DESIGN.md`](DESIGN.md) を読むこと。

タスク別の追加参照先：

- スライド：`docs/slides.md`
- 箇条書き・プレーンスライド：`docs/plain-slides.md`
- 図形・矢印・台本：`docs/diagram-shapes.md`
- 読者向け成果物と制作ログ：`docs/audience-first-redesign.md`
- 学習資料：`docs/learning-materials.md`
- 逆質問のテキストブロック：`docs/question-blocks.md`
- 画像・図解：`docs/visuals.md`
- 物語設計（冒頭・転換・見せ場・タイトル）：`docs/story-design.md`
- 資料タイプ別方針（発表資料・報告書・学習資料などの判定）：`docs/document-type-policies.md`
- AI失敗パターン集・案件間バリエーション規則：`docs/ai-failure-patterns.md`
- 品質ルーブリック（成果物・ガイド自体の到達水準）：`docs/quality-rubric.md`

## 作業時の必須事項

1. 読み手、使用場面、判断事項を明示する。
2. 事実、解釈、仮説、提案を混同しない。
3. 原材料の具体性を残す。
4. 完成版を一度に作らず、目的、原材料、構成、台本、システム、代表ページ、全体制作を分ける。
5. 対話可能な場合は、逆質問、構成案の選択、代表ページの確認で一度止まる。
6. 逆質問が必要な場合は、箇条書きで散らさず、ユーザーがそのまま回答を書き込める `text` コードブロックで提示する。
7. 逆質問では、既に分かっている前提と、こちらの仮説を先に示す。
8. 方向性の質問では、今回の題材に合わせた具体案を2〜4案提示し、推奨案と推奨理由を示す。
9. 回答欄は `推奨で進める／修正：` を基本にし、ユーザーが違うところだけ直せる形にする。
10. 空欄だけを並べて、ユーザーへ制作ブリーフをゼロから作らせない。
11. 日時、人数、予算、権利条件など、誤ると危険な事実は無理に推測しない。
12. 構成案を最低3案比較し、採用理由と不採用理由を示す。
13. 参照先を、骨格、逸脱、時間、固有素材へ分ける。
14. 参照先から借りる操作と、借りない外見を示す。
15. 同一テンプレートを内容に関係なく反復しない。
16. グリッドを同じ箱の反復ではなく、情報間の関係を作る文法として使う。
17. 静、標準、密、動の代表ページを全体制作前に確認する。
18. 独自性やダイナミックさは、秩序の上に置く一つの逸脱ルールで作る。
19. 読者向け本編、発表者用台本、付録・根拠、制作ログを分ける。
20. 制作ブリーフ、ページ役割タグ、採用・不採用理由を、読者に必要な理由なく本編へ表示しない。
21. ページ役割は、ラベルではなく内容と構造から伝える。
22. 既存成果物を再設計する場合、旧版の機能保存表を先に作る。
23. 削除した要素の理解上の機能を、新版のどこが引き継ぐか明示する。
24. 新版は旧版より、理解・判断・行動のいずれかを改善しているか比較する。
25. 抽象概念の近くに、数字、発言、操作、実物、Before／Afterなどの具体的証拠を置く。
26. 図は本文の再掲ではなく、前後、因果、差分、分岐、位置など新しい関係を示す。
27. 箇条書きで十分に伝わる並列情報を、カード、アイコン、矢印、図形へ無理に変換しない。
28. 箇条書きページを未完成扱いしない。項目の役割、文型、階層、台本との分担が成立していれば完成とみなす。
29. 図形へ変えることで新しい関係が増えない場合、プレーンな箇条書きを優先する。
30. 一つの箇条書きでは、観察、課題、行動、判断条件などの役割をそろえる。
31. 図形は名詞、矢印は動詞、囲み・レーンは文脈として定義する。
32. 矢印を置く前に、接続を「AがBへ何をする」と文章で説明する。
33. 並列、比較、グルーピングへ不要な矢印を使わない。
34. 同じ形と線種は、同じ意味へ統一する。
35. 線の交差、過剰な折れ曲がり、浮いた矢印を避ける。
36. PowerPointでは、各スライドのノート欄へ文章形式の読み上げ台本を入れる。
37. 台本には、図の読み順、主要な矢印の関係、条件、限界、前後ページへの接続を含める。
38. 台本で触れない図形や矢印は削除候補とする。
39. 箇条書きページの台本は、項目を順に読み上げるだけでなく、背景、接続、具体例、例外を補う。
40. 学習資料では、定義、識別、理由、手順、変化、失敗条件、転用を確認する。
41. 学習資料の代表ページには、手順、具体例、変化、失敗条件、判断チェックを含める。
42. 代表ページでは、30秒テスト、再話テスト、再現テスト、図単独テストを行う。
43. スマートフォン1ページ表示、PC 50％表示、A4縮小印刷で実寸閲覧を確認する。
44. 画像・図の情報上の役割を説明する。
45. 採用・棄却理由を制作ログへ残す。
46. 完成後、`docs/review-checklist.md` に沿って自己点検する。

## 対話できない場合

一度の出力を求められた場合も、内部では各ステージを順番に通す。

提出時に次を示す。

- 置いた前提
- 逆質問で提示するはずだった案、推奨案、推奨理由
- 比較した構成案
- 採用理由
- 参照した資料と借りた操作
- 読者向け本編／台本／付録／制作ログの分離
- 旧版から保存した機能と失われた機能
- グリッドと文字階層
- 図形と矢印の文法
- 箇条書きのまま残したページと、その理由
- 密度モード
- 逸脱ルール
- 代表ページで行った理解テスト
- 実寸閲覧テストの結果
- 各ページの文章形式の台本
- 捨てた案
- 未確認事項

## 保守規約

このリポジトリ自体を編集するAI・人間は、次の規約に従うこと。

1. **正典所有制**：各概念には唯一の正典ファイルがある。他ファイルは要約と参照リンクにとどめ、内容を重複させない。新しい原則を追加する前に、既存の正典ファイル（[`docs/core-principles.md`](docs/core-principles.md)、[`docs/staged-production.md`](docs/staged-production.md)、[`docs/story-design.md`](docs/story-design.md)、[`docs/visuals.md`](docs/visuals.md)、[`docs/document-type-policies.md`](docs/document-type-policies.md)、[`docs/ai-failure-patterns.md`](docs/ai-failure-patterns.md)、[`docs/quality-rubric.md`](docs/quality-rubric.md) など）に該当概念がすでにないか確認してから、追加場所を決める。
2. **DESIGN.md同期義務**：`docs/` 配下の正典ファイルを編集した場合、[`DESIGN.md`](DESIGN.md) の対応する章も同じ内容へ同期更新する。DESIGN.mdは正典群の自己完結ダイジェストであり、放置すると外部AIへ渡す版だけが古くなる。
3. **「必ず」「常に」の使用制限**：例外条件を伴わない「必ず」「常に」の追加は避ける。内容上の理由があればルールを破ってよいという本ガイドの前提と矛盾するため。
4. **追加時の重複チェック**：新しい原則やルールを追加する前に、既存ファイルに同趣旨の記述がないか検索する。見つかった場合は新規追加ではなく、既存箇所の更新または参照リンク化を優先する。
5. **案件間バリエーション規則の自己適用**：[`docs/ai-failure-patterns.md`](docs/ai-failure-patterns.md) が定める案件間バリエーション規則は、ガイド自体にも適用される。例文・数値をそのまま使い続けると、ガイドの例自体が新しい「型」になり、模倣対象が変わるだけになる。例を更新・追加する際は、題材や数値パターンの多様性を意識する。
